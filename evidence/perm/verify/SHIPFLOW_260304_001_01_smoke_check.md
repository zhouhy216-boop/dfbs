## 冒烟步骤（5–10 步）

| # | 步骤 | 预期结果 |
|---|------|----------|
| 1 | 打开 `/shipments` 列表页 | 列表正常加载，表头含：发货单号、客户、**状态**、创建时间、操作。 |
| 2 | 查看列表「状态」列 | 状态显示为中文：申请 / 备货 / 发运 / 签收完成 / 异常 / 已取消（与后端 step 一致）。 |
| 3 | 点击某行「详情」 | 右侧 Drawer 打开，展示发货详情（ID、报价单ID、类型、状态、承运商、发货日期、收货人、电话、地址、创建时间等）。 |
| 4 | 在详情 Drawer 顶部查看「当前状态」「当前步骤」 | 显示当前状态中文、当前步骤中文（来自 `GET .../workflow`）；若 workflow 接口失败，仅显示「流程信息加载失败，可刷新重试」，详情区仍可查看。 |
| 5 | 在「可执行操作」区域查看按钮 | 根据当前状态显示对应操作按钮（如 CREATED 显示 接单/标记异常/取消；PENDING_SHIP 显示 发运/标记异常/取消 等），且无多余不可用操作。 |
| 6 | 对 CREATED 状态发货单点击「接单」并确认 | 提示确认后请求成功，Drawer 内详情与「当前步骤/可执行操作」自动刷新；状态变为备货，可执行操作变为 发运/标记异常/取消。 |
| 7 | 对 PENDING_SHIP 状态点击「发运」 | 请求成功（若实体缺必填字段则后端报错属预期），详情与工作流区域刷新。 |
| 8 | 对 SHIPPED 状态点击「签收完成」并确认 | 请求成功，状态变为签收完成，可执行操作仅剩「取消」或为空。 |
| 9 | 对非完结/非取消状态点击「标记异常」 | 弹出输入框，输入异常原因后确定，请求成功，状态变为异常，可执行操作刷新。 |
| 10 | 对任意可取消状态点击「取消」 | 确认（可选填取消原因）后请求成功，状态变为已取消，可执行操作区域无按钮。 |

## 预期结果汇总

- **列表：** 能加载、状态列为 CN 且与步骤语义一致。  
- **详情：** Drawer 能打开，详情字段完整；workflow 失败时仅提示文案，不破坏详情展示。  
- **工作流：** 当前状态/当前步骤与接口一致；可执行操作与后端 `require()` 规则一致，无超范围操作。  
- **操作与刷新：** 接单/发运/签收完成/标记异常/取消 调用现有 POST，成功后详情与 workflow 自动重新拉取并更新 UI。

## 已知限制（回归观察项，供 CEO）

1. **前端全量构建：** 项目内其他文件存在既有 TypeScript 错误，`npm run build` 可能失败；本页（Shipment list + detail + workflow）在 `npm run dev` 下可验证，且 `src/pages/Shipment/index.tsx` 无 TS 报错。  
2. **发运（SHIP）：** 后端需要实体已具备必填字段（委托事项、发货日期、数量、型号、提货/收货信息、承运方等）；若从简创建的订单缺字段，点击「发运」会报错，属当前设计，Step-02 再补 per-step 必填与表单。  
3. **未找到发货单：** 访问不存在的 `id` 的 detail/workflow 时，后端可能返回 500（如 `Shipment not found`），前端会清空详情或展示错误；若需统一 404 行为，需后端全局异常映射。  
4. **操作人：** 当前前端写死 `operatorId=1`，与现有 `/shipments` 接口用法一致；权限/多操作人为后续范围。
