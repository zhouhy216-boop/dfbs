## 冒烟步骤（6–12）

| # | 步骤 | 预期结果 |
|---|------|----------|
| 1 | 打开 `/shipments`，点击某条「详情」 | Drawer 打开，展示详情与「可执行操作」按钮（含 审核并补充 / 备货确认 / 发运 / 更新物流信息 等，依当前状态）。 |
| 2 | 状态为 **CREATED** 时点击「审核并补充」 | 弹出「审核并补充」表单：收货人/联系人*、电话、收货地址*、合同号、计划发货日期、备注。 |
| 3 | 审核并补充：不填必填项直接提交 | 前端提示「请填写收货人/联系人」或「请填写收货地址」；不发起请求或请求被前端拦截。 |
| 4 | 审核并补充：填写收货人+收货地址后提交 | 请求成功；Drawer 内详情与「可执行操作」自动刷新；状态变为备货，出现「备货确认」「发运」等按钮。 |
| 5 | 状态为 **PENDING_SHIP** 时点击「备货确认」 | 弹出「备货确认」表单（数量、型号、需要包装、委托事项、提货联系人/电话、需要装车、提货地址、收货地址、备注，均可选）。提交后仅更新数据，状态仍为 PENDING_SHIP；详情与可执行操作刷新。 |
| 6 | 状态为 **PENDING_SHIP** 时点击「发运」 | 弹出「发运」表单：承运商*、物流单号*、回单URL、运单URL、备注。 |
| 7 | 发运：不填承运商或物流单号提交 | 前端提示「请填写承运商」或「请填写物流单号」；不提交或请求被前端校验拦截。 |
| 8 | 发运：填写承运商+物流单号后提交 | 请求成功；详情与可执行操作刷新；状态变为运输中（SHIPPED），出现「更新物流信息」「签收完成」等。 |
| 9 | 状态为 **SHIPPED** 时点击「更新物流信息」 | 弹出「更新物流信息」表单（物流单号、运单URL、回单URL、备注，均可选）。提交后仅更新数据，状态仍为 SHIPPED；详情与可执行操作刷新。 |
| 10 | 故意触发后端校验（如接单时实体缺收货地址、发运时缺物流单号） | 接口返回 400；前端展示对应中文提示：SHIPMENT_MISSING_RECEIVER →「请填写收货人/联系人」，SHIPMENT_MISSING_DELIVERY_ADDRESS →「请填写收货地址」，SHIPMENT_MISSING_CARRIER →「请填写承运商」，SHIPMENT_MISSING_LOGISTICS_NO →「请填写物流单号」。 |
| 11 | 对任意可操作状态执行「标记异常」或「取消」 | 行为与 Step-01 一致（原因/确认弹窗）；成功后详情与可执行操作刷新。 |
| 12 | 确认「发起售后」、列表筛选/分页、导出运单/回单 | 未被 Step-02 改动影响；功能正常。 |

## 预期结果汇总

- **每步表单存在：** CREATED → 审核并补充；PENDING_SHIP → 备货确认、发运；SHIPPED → 更新物流信息；各弹窗字段与说明一致（* 为必填仅发运/审核并补充）。  
- **最小校验生效：** 审核并补充：收货人+收货地址必填；发运：承运商+物流单号必填；前端与后端双重校验。  
- **machineCode → 中文：** 400 时根据 machineCode 显示上述 4 条中文；其他错误显示「操作失败，请稍后重试」。  
- **操作后刷新：** 任一步骤提交成功后，Drawer 内详情与「可执行操作」均重新拉取并更新，无需关抽屉再开。

## 已知限制（回归观察项，供 CEO）

1. **前端全量构建：** 项目内其他文件存在既有 TypeScript 错误，`npm run build` 可能失败；本页（Shipment list + Drawer + 步骤表单）在 `npm run dev` 下可验证，且 `src/pages/Shipment/index.tsx` 无 TS 报错。  
2. **发运 body：** 前端仅提交承运商、物流单号、回单URL、运单URL；其余字段由后端从实体补全；若实体本身缺必填信息，发运仍可能 400，需在备货/接单阶段补全。  
3. **操作人：** 前端仍使用固定 `operatorId=1`，与现有 /shipments 使用方式一致；多操作人/权限为后续范围。
