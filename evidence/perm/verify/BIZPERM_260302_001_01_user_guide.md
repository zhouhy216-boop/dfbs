# BIZPERM Step-001-01 — 业务模块视图骨架（账号编辑内）证据与用户验证指南

**Parent:** BIZPERM-260302-001-01  
**Audience:** CEO / 初学者  
**Purpose:** 说明 Step-01 交付内容：入口统一为**系统 → 账号与权限**（`/admin/account-permissions`）；在账号编辑**抽屉**内新增「**业务模块视图**」骨架（占位卡片 + 说明）；「维护目录（仅超管）」占位**仅 allowlist 超管可见**，非 allowlist 仅见说明文案；本步**无后端改动、无可写接口**。本文档支持通过 UI 完成验证。

---

## 一、前置条件

- **入口**：系统菜单 **「系统」→「账号与权限」**，路由 `/admin/account-permissions`；仅 **admin 或 super-admin** 可进入（由 `AdminOrSuperAdminGuard` 门控）。
- **allowlist 与 非 allowlist**：**allowlist**（Perm super-admin，`/v1/admin/perm/super-admin/me` 返回 allowed=true）在账号详情抽屉的「业务模块视图」骨架中可见**禁用按钮**「维护目录（仅超管）」及文案「仅超管可用（后续）」；**非 allowlist** 管理员**不显示**该按钮，仅显示一行「目录维护仅超管可用（后续）」。

---

## 二、UI 验证清单（≤10 条）

1. **打开入口**：以 admin 或 super-admin 登录，左侧菜单「系统」下点击「账号与权限」，进入 `/admin/account-permissions` 页面。
2. **打开账号详情抽屉**：在「账号」Tab 中，通过搜索得到账号列表，点击某一行或「详情」按钮，打开**账号详情抽屉**（标题含该账号用户名）。
3. **骨架位置**：抽屉内顺序为：账号信息块 → 启用/重置密码 → **业务模块视图**骨架 →（无权限时）提示 或（有权限时）保存/还原、分配角色模板、差异视图、高级编辑。业务模块视图块位于**账号信息与启用/重置之后、权限区（保存/还原与差异视图）之前**。
4. **骨架标题与说明**：骨架区块标题为「**业务模块视图**」；下方有说明：「此处将展示业务模块权限目录（按钮级）。本期为骨架占位，不影响现有权限保存/还原。」
5. **三张占位卡片**：骨架内展示三张占位卡片：**订单管理（占位）** — 权限：查看/新建/编辑/删除（占位）；**工单管理（占位）** — 权限：查看/派单/完修（占位）；**平台配置（占位）** — 权限：查看/编辑（占位）。
6. **allowlist 超管所见**：当当前用户为 allowlist 超管时，骨架底部可见**禁用**按钮「维护目录（仅超管）」及文案「仅超管可用（后续）」。
7. **非 allowlist 管理员所见**：当当前用户为非 allowlist 管理员时，骨架底部**不显示**「维护目录（仅超管）」按钮，仅显示一行「目录维护仅超管可用（后续）」。
8. **现有保存/还原与差异视图未变**：抽屉内保存、还原、分配角色模板、差异视图（继承/追加/移除/最终生效）、高级编辑（折叠）的布局与行为与 Step-01 实施前一致；骨架为独立区块，不依赖、不修改上述状态与 DOM。
9. **入口门控**：非 admin 用户不应看到「账号与权限」菜单项；直接访问 `/admin/account-permissions` 应被重定向或拦截。
10. **无新接口**：本步未新增任何可写或只读接口；骨架为纯占位 UI，维护目录按钮为禁用占位，无点击请求。

---

## 三、说明与约束

- **本期为骨架**：业务模块视图区块为占位，不绑定真实数据与接口；后续步骤再接入业务模块目录与维护能力。
- **无后端变更**：Step-01 未修改后端；未暴露任何新的可写 API；未来目录维护接口须在服务端做 allowlist 门控（见代码内 TODO(BIZPERM) 注释）。

---

## 四、CEO 一句话验证要点

1. **入口与骨架**：在「系统 → 账号与权限」→ 账号 Tab 中，点击某账号打开详情抽屉；在账号信息与启用/重置下方、权限区上方可见「业务模块视图」骨架（三张占位卡片 + 说明）。
2. **超管 vs 普通管理员**：allowlist 超管在骨架底部能看到禁用按钮「维护目录（仅超管）」；非 allowlist 管理员**看不到**该按钮，只看到「目录维护仅超管可用（后续）」。
3. **无回归**：保存/还原、差异视图、分配角色模板等原有流程与交互保持不变；本步未引入任何新接口。
