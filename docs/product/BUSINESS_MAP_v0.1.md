## 0) 核心事实（统一口径）
- 设备号 = 机器号（同一个东西）。
- 客户自带平台/SIM：合同里不体现；公司提供的平台/SIM：合同录入时会体现（平台类型如映翰通、支付宝刷脸等）。
- 平台成本：你导入，系统用于报表核算（暂不对接外部系统）。
- 保修期：可能从【发货时间 / 签收时间 / 调试完成时间】起算；影响保内/保外与收费判断。
- 报价：四类收费统一成一种“报价单对象”，用“报价类型”区分。

## 3) 端到端流程地图（把业务主线落到“模块锚点/页面锚点”）

### P1 合同 → 评审 → 生产准备 → 交付 → 签收关闭（最长主线）
1. 销售签合同 & 录入合同：M01（合同 `/master-data/contracts`）:contentReference[oaicite:13]{index=13}  
2. 营业企画评审/补充：M02（暂无页面锚点，待补）  
3. 生产企画安排生产、生成设备号：M03（暂无页面锚点，待补）  
4. 物流安排交付：M04（发货 `/shipments`）:contentReference[oaicite:14]{index=14}  
5. 仓库出库/相关库存动作：M07（库存 `/warehouse/inventory`）:contentReference[oaicite:15]{index=15}  
6. 客户签收 → 交付关闭：目前视为 M04 的流程结果（细节后续在 M04 模块详细讨论）

### P2 入网准备协作（你新增的关键链路）
> 时机：营业企画下达生产准备时同步触发；产物：设备 ↔ SIM ↔ 平台 的绑定关系 + SIM开通 + 平台侧绑定/新增平台  
1. 营业企画：把“平台信息 + SIM卡号”发给生产企画和平台主管：M02（暂无页面锚点，待补）  
2. 生产企画：把 SIM卡/平台信息 与 设备号绑定：M03（暂无页面锚点，待补；可能落在“机器详情”）  
3. 平台主管：SIM开通 & 平台绑定/新增平台：M08（`/platform/applications`、`/platform/applications/sim-activation` 等页面族）:contentReference[oaicite:16]{index=16}  
4. 结果：设备达到“入网准备完成”状态（具体状态字段在后续 M08/M03 详细讨论中定义）

### P3 售后工单闭环（维修主线）
- 客户入口：`/public/repair` 发起报修（无需登录）:contentReference[oaicite:17]{index=17}  
- 内部处理：`/work-orders` 列表与 `/work-orders/:id` 详情（需要 `work_order:VIEW`）:contentReference[oaicite:18]{index=18}  
- 典型动作（已体现在现有页面能力描述中）：创建/拒绝/接单/派单/记录/配件/签收/完结等:contentReference[oaicite:19]{index=19}  
- “新机培训”：作为工单类型复用 P3，只是发起入口/发起人不同（销售发起 → 营业企画审核 → 客服受理 → 服务经理执行 → 回访闭环）

### P4 运输异常闭环（从交付中间节点发起）
- 入口：`/after-sales` 列表、`/after-sales/:id` 详情:contentReference[oaicite:20]{index=20}  
- 关键动作：提交/接收/处理/退回/完成:contentReference[oaicite:21]{index=21}  
- 口径：系统不自动识别异常，由物流人员标记/录入异常（细节在 M04/M运输异常模块详细讨论中定）

### P5 四类收费闭环（统一报价对象）
- 报价入口统一在 `/quotes`：列表/创建/编辑/确认/作废/提交/导出等:contentReference[oaicite:22]{index=22}  
- 财务入口 `/finance`（支付/费用/对账/对账单等）:contentReference[oaicite:23]{index=23}  
- 四类场景（统一对象 + 不同触发方）：
  1) 平台费&SIM费续费：平台主管发起
  2) 委托运输收费：物流主管发起
  3) 上门维修收费：客服代表发起（保外/保内人为）
  4) 配件收费：客服代表发起 → 仓库发货
- 共通节点：客户确认（服务经理）→ 会计确认报价 → 收款 → 会计确认到账 → 执行（续费/运输/维修/发货）

## 4) 通用规则（避免“最长流程”之外的情况漏掉）
- 允许从中间节点发起：例如运输异常、配件需求、续费等，不需要从合同起步。
- 通用动作层：编辑/提交/审批/拒绝/分配/关闭/删除/导出等动作属于“对象生命周期”，在逐模块详细讨论时把每个对象的“状态 + 动作”写清即可。
- 外部系统：暂不对接；可以先通过“导入/记录/附件归档”满足报表核算与追溯。
