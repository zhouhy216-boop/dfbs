# CHAIN_25_SERVICE_SUPPLEMENT_AGREEMENT_LIFECYCLE_BRANCH_v1

## 1. 文档定位
本链用于说明服务补充合同 / 协议的生命周期管理口径。

其核心不是购销合同评审，也不是工单执行过程，而是解决以下问题：
- 是否存在服务补充合同 / 协议
- 当前是否处于生效状态
- 若同一客户主体存在多份同类协议，当前应取哪一份
- 当前有效协议会影响哪些下游业务

本链的目标，是将服务部侧的服务补充合同 / 协议，从主数据中的说明性内容，提升为一条可独立理解和管理的业务支链。

## 2. 核心业务目标
服务补充合同 / 协议管理的核心目标，是对服务部与客户之间围绕售后服务、平台 / SIM 服务及专项服务形成的补充约定，建立统一的生命周期管理口径，使系统能够明确：
- 当前有哪些服务补充合同 / 协议
- 哪些协议当前有效
- 哪些协议已失效或关闭
- 多份协议并存时当前应以哪份为准
- 当前有效协议会对哪些后续业务产生影响

本链强调的是服务约定的持续有效性判断和业务引用基础，而不是合同评审本身。

## 3. 管理对象
本链的管理对象为服务补充合同 / 协议。

服务补充合同 / 协议区别于购销合同，属于服务部侧围绕客户服务关系形成的补充约定。其主要作用不是承接销售成交本身，而是承接后续服务中的价格、时效、平台 / SIM 管理和专项服务安排等业务约定。

因此，本链只管理服务补充合同 / 协议，不管理购销合同评审过程。

## 4. 协议分类
当前阶段，服务补充合同 / 协议统一按以下三类管理：

1. 维修协议  
用于约定维修服务相关内容，包括但不限于服务范围、服务价格、服务时效等。

2. 平台 / SIM 协议  
用于约定平台服务、SIM 服务及相关收费、限免、服务支持等内容。

3. 专项服务协议  
用于约定面向特定项目、活动或专题服务的专项服务内容。

当前版本不再继续细分更多协议类型，以避免生命周期管理口径过早复杂化。

## 5. 协议归属与适用范围
服务补充合同 / 协议由服务部侧管理，不归入购销合同评审链处理。

本链适用范围包括：
- 已与客户形成服务关系后，围绕维修、平台 / SIM、专项服务形成的补充约定
- 对后续服务执行、服务价格、平台 / SIM 建议值、工单时效考核存在影响的服务类协议

本链不适用于：
- 购销合同本身
- 纯销售成交条件
- 法务合同模板管理
- 线下盖章流转细节

## 6. 当前生效判断
服务补充合同 / 协议是否当前生效，是本链的核心判断。

当前阶段，按以下原则判断：

1. 协议存在明确起止时间，且当前时间处于有效期间内的，视为当前生效。

2. 协议虽无严格起止区间，但未触发失效或关闭条件的，也可视为当前生效。

3. 维修协议、平台 / SIM 协议，即使已超约定期间但尚未续签，当前仍默认继续按原协议执行，除非已有新的替代协议或明确失效处理。

4. 专项服务协议在专项服务结束后关闭，不默认延续。

因此，本链中的“当前生效”不等于“严格未过合同日期”，而是应结合协议类型和业务默认延续规则综合判断。

## 7. 默认延续与关闭规则
不同类型的服务补充合同 / 协议，在生命周期上采用不同的延续与关闭规则。

### 7.1 维修协议
维修协议在超期未续签时，默认继续按原协议执行。  
其目的，是避免客户已持续接受维修服务时，因为协议续签动作滞后，导致服务口径和业务执行失去依据。

### 7.2 平台 / SIM 协议
平台 / SIM 协议在超期未续签时，也默认继续按原协议执行。  
其目的，是保持平台服务、SIM 服务及相关建议值和收费建议口径的连续性，避免因协议到期但未及时续签而造成业务中断或口径漂移。

### 7.3 专项服务协议
专项服务协议在专项服务结束后关闭。  
专项服务协议不采用默认延续逻辑，因为其本质通常是围绕特定项目、阶段或专题服务形成的临时性约定，服务完成后即应结束其业务效力。

## 8. 多协议并存与冲突处理
同一客户主体下，原则上不应存在“约定项相同”的多份当前生效服务补充合同 / 协议。

但在现实业务中，仍可能出现：
- 历史协议未及时失效
- 新协议已签订但旧协议未关闭
- 相同约定项存在多份并行记录

当前阶段对此的统一口径为：

1. 原则上，同一客户主体不应存在约定项相同的多份当前生效协议。

2. 如现实中已经存在，则系统建议取签订时间更晚的一份，作为当前建议依据。

3. 当前版本不展开“多份不同约定项同时生效时如何组合取值”的复杂细则。

本链当前先解决“同类冲突时优先取哪份”的基本问题，不进一步展开更复杂的条款级组合引用逻辑。

## 9. 对下游业务的影响
当前有效的服务补充合同 / 协议，会影响多个后续业务链，但当前阶段重点只冻结以下三类影响。

### 9.1 平台 / SIM 建议值
若存在当前生效的平台 / SIM 协议，则平台相关、SIM 相关的建议值应优先按该协议承接。  
这里强调的是“建议值优先依据”，而不是强制锁死所有后续业务结果。

### 9.2 工单服务时效引用依据
若当前生效协议中存在响应、到场、完修等服务时效约定，则后续工单时效考核应可引用该协议中的服务约定作为判断依据。

本链只负责提供“当前有效服务约定”，不负责具体工单是否达标的判断。

### 9.3 服务价格类建议依据
若当前生效协议中存在服务价格、配件价格等约定，则后续业务可据此作为服务价格类建议依据。

当前版本仅冻结“可作为建议依据”这一层，不展开价格优先级、复杂覆盖关系和最终执行细则。

## 10. 与现有业务链的边界
### 10.1 与 CHAIN_01 的边界
CHAIN_01 管理购销合同评审。  
本链管理服务补充合同 / 协议生命周期。  
两者同属合同类信息，但业务目标不同，不应混写。

### 10.2 与 CHAIN_13 的关系
CHAIN_13 负责主数据总口径。  
本链将其中“服务补充合同 / 协议”的内容单独展开为一条业务支链，用于明确生命周期判断和下游引用边界。

### 10.3 与 CHAIN_06 的边界
CHAIN_06 负责工单执行过程。  
本链不管理工单处理动作，只向工单链提供“当前有效的服务约定”作为引用依据。

### 10.4 与未来 SLA 履约考核链的关系
本链是未来“SLA 履约与工单时效考核链”的上游。  
本链负责回答“当前有效协议是谁”；未来 SLA 链负责回答“工单按该协议是否达标”。

## 11. 当前边界与不展开项
本链当前只定义服务补充合同 / 协议的生命周期管理口径，暂不展开以下内容：

- 不展开购销合同评审
- 不展开法务盖章、线下签章过程
- 不展开合同模板管理
- 不展开审批流设计
- 不展开附件管理细节
- 不展开页面、字段、权限、提醒实现
- 不展开多份不同约定项同时生效时的复杂组合取值细则
- 不展开工单时效统计、达标率分析与 SLA 预警

本链只解决“服务补充合同 / 协议如何判断当前有效并影响下游”的背景问题，不延伸到制度设计和系统实现细节。

## 12. 小结
服务补充合同 / 协议生命周期管理，是服务部侧服务约定管理的独立业务支链。

其核心价值不在于记录一份协议存在，而在于持续判断：
- 当前哪份协议有效
- 当前应以哪份协议为准
- 当前有效协议会影响哪些后续业务

当前阶段，本链已冻结以下核心原则：
- 仅管理服务补充合同 / 协议，不管理购销合同
- 分类仅保留维修协议、平台 / SIM 协议、专项服务协议三类
- 维修协议、平台 / SIM 协议可默认继续执行
- 专项服务协议服务结束即关闭
- 同一客户主体若存在约定项相同的多份当前生效协议，建议取签订时间更晚的一份
- 当前有效协议主要影响平台 / SIM 建议值、工单服务时效引用依据和服务价格类建议依据
