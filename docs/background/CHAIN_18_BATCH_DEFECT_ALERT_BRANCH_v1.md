## CHAIN_18_BATCH_DEFECT_ALERT_BRANCH_v1.md

## 1. 链条定位
批量不良预警不是售后工单主链本身，而是工单侧的一条风险识别支链。

售后工单主链解决的是：
报修 → 受理 → 派单 → 处理 → 完修 → 回访 → 关闭

批量不良预警解决的是：
基于工单中的维修与换件事实，识别某机型是否可能出现了集中性、批量性的零部件异常风险。

说明：
- 工单主链的目标，是完成一次服务处理
- 批量不良预警的目标，是从多笔工单中识别风险趋势
- 因此，批量不良预警依赖工单，但不替代工单
- 批量不良预警属于工单侧支链，不属于独立主交易流程

---

## 2. 本链条的业务目的
建立批量不良预警的主要目的，不是单纯统计换件次数，而是尽早发现某机型是否出现值得关注的集中性问题。

当前主要目标包括：

1. 识别同一机型是否在短期内重复更换同一零部件
2. 区分单机个体问题与可能的批量性问题
3. 为客服提供需要重点关注的风险提示
4. 为后续质量分析、机型复盘、供应商问题排查提供基础
5. 为后续统计某机型高风险零部件提供整理基础

说明：
- 当前阶段首先强调风险识别价值
- 当前阶段不把批量不良预警定义成完整质量管理流程
- 当前阶段也不把批量不良预警定义成供应商责任判定系统

---

## 3. 批量不良预警与工单主链的关系
批量不良预警的数据来源于工单，但工单与批量不良预警不是同一个对象。

当前冻结口径为：

1. 工单负责记录一次真实服务事件
2. 批量不良预警负责从多笔工单中识别集中性风险
3. 工单中的保内/保外判断、是否人为、是否换件、换了什么零部件，都是预警的来源
4. 不是所有工单都进入批量不良预警统计范围
5. 批量不良预警是在工单事实基础上的规则识别结果

说明：
- 工单是原始事实
- 批量不良预警是规则识别结果
- 因此，批量不良预警不是对工单的简单复制

---

## 4. 批量不良预警与故障库的关系
批量不良预警与故障库有关联，但两者不是同一对象。

当前冻结口径为：

1. 故障库用于沉淀故障知识与处理经验
2. 批量不良预警用于识别集中性风险
3. 故障库解决“类似问题怎么处理”
4. 批量不良预警解决“这个机型是否可能存在批量问题”
5. 两者都来源于工单，但业务目的不同

说明：
- 故障库偏知识沉淀
- 批量不良预警偏风险识别
- 当前阶段不把两者混成一个模块

---

## 5. 批量不良预警的判断基础
当前阶段，批量不良预警不以自然语言故障现象作为主判断依据，而以换件事实作为主要判断基础。

当前冻结口径为：

1. 批量不良预警优先基于维修工单中的实际换件记录
2. 不以自然语言故障现象作为第一版主要触发依据
3. 自然语言故障现象更适合用于故障库归类
4. 换件事实更适合用于批量不良预警判断

说明：
- 自然语言故障现象存在表述差异
- 换件记录相对更稳定、更适合作为规则判断基础
- 因此当前阶段先采用“换件事实优先”的原则

---

## 6. 统计范围口径
当前阶段，进入批量不良预警统计范围的工单事件，应同时满足以下条件：

1. 属于维修类工单
2. 属于保内设备
3. 属于非人为原因
4. 实际发生了零部件更换

当前不纳入统计范围的情况包括：

1. 保外维修
2. 保内但判定为人为损坏
3. 未发生换件，仅调试、清洁、参数处理等情况
4. 培训、巡检、移机等非维修类工单
5. 其他不以维修换件为核心事实的工单场景

说明：
- 批量不良预警当前只关注“保内 + 非人为 + 换件”的维修事实
- 这样可以尽量把产品/机型风险与客户使用问题、人为损坏、一般维护行为区分开

---

## 7. 归集维度口径
当前阶段，批量不良预警按以下维度归集：

机型 + 零部件

当前冻结口径为：

1. 预警不是只按零部件全局统计
2. 预警也不是只按故障现象统计
3. 预警应先看“某机型是否集中更换某零部件”
4. 同一零部件若被多个机型共用，当前阶段不直接按全局共件异常统一预警
5. 当前阶段先按机型维度收敛风险判断

说明：
- 同一零部件可能被多个机型共用
- 因此不能简单按“某零部件全局更换次数”直接预警
- 当前阶段先聚焦“某机型 + 某零部件”的风险关系

---

## 8. 零部件识别口径
批量不良预警中的零部件识别，沿用主数据中的统一识别规则。

当前冻结口径为：

1. 有图号的零部件，优先按图号识别
2. 无图号的零部件，按名称 + 供应商识别
3. 当前阶段不为批量不良预警额外建立独立零部件识别口径
4. 批量不良预警应复用零部件主数据已冻结的识别规则

说明：
- 这样可以保证预警口径与零部件主数据口径一致
- 避免同一个零部件在不同场景下出现不同识别方式

---

## 9. 计数单位口径
当前阶段，批量不良预警的计数单位，不按单纯换件次数计数，而按不同设备数计数。

当前冻结口径为：

1. 同一机型、同一零部件在观察窗口内，应按“涉及多少台不同设备”来统计
2. 不按“总共换了多少次”直接作为第一计数单位
3. 同一台设备重复更换同一零部件，不直接视为批量不良依据
4. 同一台设备的重复换件，更偏向单机个体问题或返修问题
5. 批量不良预警更关注“是否扩散到多台不同设备”

说明：
- 单机重复换件，不一定说明该机型存在批量问题
- 多台不同设备在相近时期重复更换同一零部件，更能说明存在集中性风险

---

## 10. 时间窗口与阈值口径
当前阶段，批量不良预警采用滚动时间窗口进行判断。

当前冻结口径为：

1. 默认采用滚动 90 天窗口
2. 在滚动 90 天内进行统计判断
3. 同一机型下，若 3 台不同保内设备更换同一零部件，则触发批量不良预警
4. 当前阶段不再使用“连续三次”作为主规则表述
5. 当前阶段先采用“90 天 + 3 台不同设备”的口径

说明：
- “连续三次”容易把单机重复返修误判成批量不良
- “90 天 + 3 台不同设备”更适合作为第一版稳妥规则
- 当前阶段先求低误报、可解释，不追求过高灵敏度

---

## 11. 排除项口径
当前阶段，以下情况不纳入批量不良预警：

1. 消耗品
2. 明显不属于质量风险的辅料
3. 人为损坏导致的换件
4. 单机重复更换同一零部件但未扩散到其他设备的情况
5. 其他不宜视为机型批量风险的换件情况

说明：
- 批量不良预警应尽量聚焦可能的机型质量风险
- 不应被消耗品、辅料、人为损坏等情况污染结果

---

## 12. 提醒对象口径
当前阶段，批量不良预警的提醒对象先定为客服。

当前冻结口径为：

1. 预警触发后，当前先提醒客服
2. 当前阶段不展开更复杂的多部门联动提醒
3. 后续如有需要，可再扩展到其他角色
4. 当前版本先不展开预警后的详细升级流转机制

说明：
- 当前先保证“有人看到预警”
- 后续是否扩展到服务经理、质量角色、管理角色，留待后续专题讨论

---

## 13. 证据包口径
批量不良预警不能只给出一个抽象提示，而应能够追溯触发依据。

当前冻结口径为：

1. 每条预警都应能看到对应机型
2. 每条预警都应能看到对应零部件
3. 每条预警都应能看到统计时间范围
4. 每条预警都应能看到涉及的设备号
5. 每条预警都应能看到对应工单来源
6. 每条预警都应能看到换件事实
7. 当前阶段不要求展开更复杂的质量结论，只要求能够说明“为什么会触发预警”

说明：
- 当前阶段的重点，是让预警“有依据、可解释”
- 不是只给出一个无法追溯来源的风险提示

---

## 14. 与机型资料的关系
批量不良预警需要和机型资料、故障库一起按机型聚合查看，但三者不是同一个对象。

当前冻结口径为：

1. 机型资料包括视频、PPT等资料
2. 故障库用于沉淀该机型常见故障与解决经验
3. 批量不良预警用于提示该机型是否存在集中性零部件风险
4. 三者共享同一个机型视角
5. 但三者不是同一个对象

说明：
- 机型资料解决“了解这个机型”
- 故障库解决“这个机型常见问题怎么处理”
- 批量不良预警解决“这个机型最近是否出现值得关注的集中性异常”

---

## 15. 当前边界
当前阶段，本链条只冻结第一版批量不良预警规则，不展开以下内容：

1. 更高等级预警的分级规则
2. 跨机型共件异常的统一预警规则
3. 预警后的审批、结案、责任认定流程
4. 页面、字段、按钮、报表实现方式
5. 更复杂的算法优化方式
6. 供应商责任、质量责任的正式判定机制

说明：
- 当前先把第一版预警口径定义清楚
- 后续如需扩展高等级预警或跨机型分析，可再单独展开
- 当前版本不把预警后的完整管理流程一并写入

---

## 16. 当前冻结结论
当前可冻结的总口径为：

- 批量不良预警属于工单侧的一条风险识别支链
- 批量不良预警不替代工单主链
- 工单是原始事实来源，批量不良预警是规则识别结果
- 批量不良预警与故障库有关联，但不是同一个对象
- 当前阶段，批量不良预警优先基于换件事实判断
- 当前阶段，不以自然语言故障现象作为第一版主要触发依据
- 进入预警统计范围的事件，应同时满足：维修类工单、保内、非人为、实际发生换件
- 批量不良预警按“机型 + 零部件”归集
- 有图号零部件优先按图号识别
- 无图号零部件按名称 + 供应商识别
- 批量不良预警按不同设备数计数，不按单纯换件次数计数
- 同一台设备重复更换同一零部件，不直接视为批量不良依据
- 当前默认采用滚动 90 天窗口
- 同一机型下，若 90 天内有 3 台不同保内设备更换同一零部件，则触发批量不良预警
- 当前阶段不再使用“连续三次”作为主规则表述
- 消耗品、辅料、人为损坏导致的换件，不纳入批量不良预警
- 批量不良预警当前先提醒客服
- 每条预警都应能追溯到对应机型、零部件、设备号、工单来源和换件事实
- 批量不良预警需要与机型资料、故障库一起按机型聚合查看
- 机型资料、故障库、批量不良预警共享机型视角，但不是同一个对象
- 当前阶段暂不展开高等级预警、跨机型共件异常、预警后审批与责任认定流程
