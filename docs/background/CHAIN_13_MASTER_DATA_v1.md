## 2. 主数据整体定位

### 2.1 主数据的作用
主数据用于承接全系统共用的基础对象，为合同、发货、运输异常、工单、报价、结算、仓库、平台费、SIM 费等模块提供统一基础资料。

当前已确认的主数据对象包括：
1. 客户
2. 合同
3. 机器
4. SIM
5. 平台
6. BOM / 零部件
7. 机器关键零部件 SN 主数据

### 2.2 主数据与业务单据的关系
当前冻结口径为：

1. 主数据不等于业务单据
2. 主数据是业务单据的上游基础对象
3. 合同、发货、工单、报价、结算、仓库等业务对象都不替代对应主数据对象
4. 主数据的核心作用包括：
   - 统一识别
   - 联想输入
   - 引用
   - 关系追溯
   - 历史留痕

### 2.3 客户口径的默认解释
当前冻结口径为：

1. 客户主数据需要区分：
   - 客户主体
   - 客户业务单位
2. 合同默认挂在**客户主体**下
3. 除合同外，机器、SIM、平台账号等后续主数据章节中出现的“客户”，默认指**客户业务单位**
4. 若个别场景不是客户业务单位，应在对应章节单独说明

### 2.4 主数据录入与确认总规则
当前冻结口径为：

1. 当前所有主数据中，除 **ICCID** 外，其他主数据都允许录入主数据中暂不存在的数据
2. 主数据的主要作用是：
   - 联想输入
   - 引用
3. 若业务人员输入了主数据中不存在的数据，则进入**数据确认中心**
4. 由管理员在数据确认中心判断：
   - 这是一个新数据
   - 还是原有数据被业务人员填写错误
5. 因此除 ICCID 外，主数据不要求业务录入时必须百分之百预先完备
6. 但最终仍应通过数据确认中心回收到统一主数据口径

## 3. 客户主数据

### 3.1 客户定位
客户主数据用于管理客户基础身份信息、客户归属关系以及客户在系统中的统一识别口径。

客户主数据的目标包括：
1. 统一客户识别
2. 支撑合同录入
3. 支撑报价录入
4. 支撑搜索联想
5. 承接客户归属管理
6. 为后续业务对象提供统一客户基础

### 3.2 客户唯一识别口径
当前冻结口径为：

1. 客户主键使用客户代码
2. 客户代码不是系统自动生成
3. 客户代码是公司内部管理代码
4. 客户代码需要手动输入
5. 系统以客户代码而不是客户名称识别客户

### 3.3 客户名称口径
当前冻结口径为：

1. 客户名可能是企业名称，也可能是个人名称
2. 因为存在个人客户，所以客户名称可能重复
3. 客户名称不作为唯一主键
4. 系统不应按客户名称做唯一限制

### 3.4 简称 / 别称口径
当前冻结口径为：

1. 同一客户可以存在多个简称
2. 同一客户可以存在多个别称
3. 简称 / 别称不仅用于展示
4. 简称 / 别称需要参与搜索与联想输入

适用场景包括：
- 客户搜索
- 合同录入
- 报价录入

目标是：
- 当业务人员输入简称或别称时，系统可辅助联想到对应客户主名

### 3.5 客户归属口径
当前冻结口径为：

1. 客户归属到营业本部 - 营业统括部下的事业部
2. 客户主档只有一个当前归属事业部
3. 客户归属允许换绑
4. 换绑后，客户主档更新为新的当前归属事业部
5. 历史归属需要保留，不覆盖历史记录

说明：
- 客户当前归属与历史归属应区分
- 客户换绑不等于历史业务全部改挂新归属

### 3.6 客户状态口径
当前冻结口径为：

1. 客户建立后长期保留
2. 不做删除式管理
3. 客户主档需要有状态栏
4. 当前阶段只冻结“存在状态栏”这一原则
5. 状态名称与流转规则后续再定

### 3.7 客户与其他业务对象关系
当前冻结口径为：

1. 同一客户可以签订多个合同
2. 客户与合同是一对多关系
3. 同一客户也可以对应多个报价单
4. 同一客户也可以对应多个工单
5. 同一客户也可以对应多个发货记录
6. 同一客户也可以对应多个结算记录

说明：
- 客户主数据是这些业务对象的上游基础对象
- 合同、报价、工单、发货、结算都不替代客户主档

### 3.8 重复客户提醒口径
当前冻结口径为：

1. 因客户名称可能重复，系统不应简单按名称禁止录入
2. 对疑似重复客户，应做提醒
3. 由业务人员确认是否为同一客户
4. 是否新建，由业务人员结合客户代码和实际情况判断

### 3.9 全国性客户管理建议
当前冻结口径为：

对于全国性客户，不建议简单拆成多个彼此独立的重复客户主档。

建议区分两层口径：

#### 3.9.1 客户主体
客户主体用于承接：
- 客户代码
- 正式客户名
- 简称 / 别称
- 当前归属事业部
- 合同主体关系

客户主体通常对应总客户 / 签约主体。

#### 3.9.2 客户业务单位
客户业务单位用于承接：
- 分公司
- 事业部
- 使用单位
- 报修单位
- 费用承担单位
- 局部结算单位

客户业务单位不等于新的客户主体，
而是客户主体下面的业务承接单元。

### 3.10 全国性客户下的业务关系口径
当前冻结口径为：

1. 全国性客户通常由总公司或统一主体签订合同
2. 合同通常挂在客户主体下
3. 售后报修可按客户业务单位发起
4. 部分费用可按客户业务单位单独处理
5. 不应通过重复建立多个客户主体来表达同一全国性客户的多业务单位场景

### 3.11 当前边界
当前阶段只冻结以下内容：

1. 客户代码作为主键
2. 客户名称允许重复
3. 简称 / 别称可用于搜索与录入联想
4. 客户主档只有一个当前归属事业部，但允许换绑
5. 历史归属保留
6. 客户与合同是一对多关系
7. 客户建立后长期保留，但有状态栏
8. 疑似重复客户要提醒，由业务人员确认
9. 全国性客户建议区分客户主体与客户业务单位两层口径

当前阶段暂不展开：
- 具体状态名称
- 客户业务单位的详细字段
- 客户与联系人、地址、开票资料的细节结构

### 3.12 当前已冻结的关键原则
1. 客户主键使用客户代码
2. 客户代码由公司内部管理，手动输入
3. 客户名称可能重复，不作为唯一识别口径
4. 同一客户可存在多个简称 / 别称
5. 简称 / 别称参与搜索、合同录入、报价录入联想
6. 客户归属到营业本部 - 营业统括部下的事业部
7. 客户主档只有一个当前归属事业部
8. 客户归属允许换绑
9. 历史归属需要保留
10. 同一客户可签订多个合同
11. 客户与合同是一对多关系
12. 客户建立后保留，不删除，但有状态栏
13. 对疑似重复客户应做提醒，由业务人员确认
14. 全国性客户不建议拆成多个重复客户主档
15. 全国性客户建议区分客户主体与客户业务单位两层口径
16. 合同通常挂在客户主体下
17. 售后报修和部分费用处理可落到客户业务单位

## 4. 合同主数据

### 4.1 合同定位
合同主数据用于承接客户与公司之间的正式约定，为设备保修、平台费、SIM 卡费用、售后服务时效、专项服务、报价、发货、工单、结算等业务提供可追溯依据。

说明：
- 合同主数据不等于客户主数据
- 合同默认挂在客户主体下
- 同一客户主体可对应多个合同

### 4.2 合同分类口径
当前冻结口径为：

合同分为两类：

1. **购销合同**
2. **服务补充合同 / 协议**

说明：
- 当前阶段区分“购销合同”和“服务补充合同 / 协议”两类
- 不单独再拆一个独立的“协议主数据对象”

### 4.3 合同归属口径
当前冻结口径为：

1. 购销合同归属**销售**
2. 服务补充合同 / 协议归属**服务部**

### 4.4 购销合同口径
当前冻结口径为：

购销合同主要对应成品机销售。

购销合同一般也可能约定：
1. 保修期
2. 平台相关内容
3. SIM 卡相关内容

补充说明：
- 购销合同一般**没有到期时间**
- 同一客户主体可能存在多个历史购销合同

### 4.5 服务补充合同 / 协议口径
当前冻结口径为：

服务补充合同 / 协议分为三类：

#### 4.5.1 售后维修合同 / 协议
用于约定售后维修相关内容，例如：
- 响应时效
- 到场时效
- 完修时效
- 配件价格
- 服务价格

#### 4.5.2 平台费 / SIM 卡协议
用于约定：
- 平台服务费用
- SIM 卡服务费用

#### 4.5.3 专项服务合同 / 协议
用于约定某次专项服务，例如：
- 改造
- 检修
- 其他一次性专项服务

补充说明：
- 本节只冻结服务补充合同 / 协议在主数据中的分类口径
- 服务补充合同 / 协议的当前有效判断、默认延续与关闭规则，以 CHAIN_25 为准
- 本文不再展开服务补充合同 / 协议的完整生命周期规则

### 4.6 客户与合同关系
当前冻结口径为：

1. 同一客户主体可以签订多个合同
2. 客户主体与合同是一对多关系
3. 合同作为客户主体下的业务约定依据，不替代客户主档

### 4.7 设备与合同关系
当前冻结口径为：

1. 设备保修期建议值按**关联的合同号**走
2. 发货信息中存在合同号，因此设备可追溯到具体合同
3. 当同一客户主体存在多个购销合同时，设备保修判断应优先依据该设备关联的合同号

### 4.8 平台 / SIM 建议值口径
当前冻结口径为：

平台费与 SIM 卡费用在系统中只提供建议值，不作为强制锁死规则。

建议值规则如下：

1. 先判断该客户主体是否存在当前有效的服务补充合同 / 协议
2. 若存在当前有效的服务补充合同 / 协议，则按该服务补充合同 / 协议给出平台 / SIM 建议值
3. 若不存在当前有效的服务补充合同 / 协议，则按同一客户主体历史购销合同最低价给出建议值

说明：
- 该建议值规则只影响平台 / SIM 收费建议值，不自动覆盖原合同事实
- 最终是否采用该建议值，由相关业务员结合实际业务判断
- 该判断仅限平台 / SIM 收费建议值场景，不延伸到协议是否当前有效、主数据事实或 SLA 考核口径

### 4.9 合同冲突与并存时的平台 / SIM 建议值口径
当前冻结口径为：

若同一客户主体的购销合同与服务补充合同 / 协议在平台费、SIM 卡费用建议值上存在差异，则：

1. 系统建议值优先取当前有效的服务补充合同 / 协议
2. 该优先级仅影响平台 / SIM 收费建议值
3. 不自动覆盖购销合同原始事实
4. 最终是否采用该建议值，由相关业务员决定
5. 该决定仅限平台 / SIM 收费建议值，不延伸到协议是否当前有效、主数据事实或 SLA 考核口径

对于同一客户主体的服务补充合同 / 协议，当前冻结口径为：

1. 原则上不应存在约定项相同的多份当前有效服务补充合同 / 协议
2. 若现实中存在，则系统建议值取签订时间更晚的那一份作为当前建议依据
3. 上述规则仅用于系统建议值，不等于自动覆盖原合同事实

### 4.10 服务合同时效口径
当前冻结口径为：

1. 服务补充合同 / 协议的当前有效判断、默认延续与关闭规则，以 CHAIN_25 为准
2. 若当前有效服务补充合同 / 协议中存在响应、到场、完修等服务时效约定，则维修工单优先引用该约定进行考核
3. 若不存在可引用的当前有效服务时效约定，则回落公司默认标准
4. 服务时效定义及考核边界以 CHAIN_26 为准

补充说明：
- 这里的服务时效，是指售后服务中的响应时效、到场时效、完修时效等约定
- 本节只冻结合同主数据与工单时效考核之间的引用关系
- 本文不再展开服务补充合同 / 协议的生命周期细节，也不再展开工单时效考核的定义细节

### 4.11 合同主数据当前边界
当前阶段只冻结以下内容：

1. 合同分为购销合同、服务补充合同 / 协议两类
2. 购销合同归属销售
3. 服务补充合同 / 协议归属服务部
4. 服务补充合同 / 协议分为维修、平台 / SIM、专项服务三类
5. 设备保修期建议值按关联合同号走
6. 发货信息可作为设备与合同的关联依据
7. 平台 / SIM 费用只给系统建议值，不锁死
8. 若有当前有效服务补充合同 / 协议，则平台 / SIM 建议值按其给出
9. 若无当前有效服务补充合同 / 协议，则平台 / SIM 建议值按历史购销合同最低价给出
10. 购销合同与服务补充合同 / 协议在平台 / SIM 建议值上存在差异时，建议值优先取当前有效服务补充合同 / 协议
11. 服务补充合同 / 协议的当前有效判断、默认延续与关闭规则，以 CHAIN_25 为准
12. 服务时效与维修工单考核的连接关系，以 CHAIN_26 为准
13. 若存在多份约定项相同的当前有效服务补充合同 / 协议，则取签订时间更晚的一份作为当前建议依据

当前阶段暂不展开：
- 合同编号规则
- 合同附件管理
- 合同审批流程
- 多份不同约定项同时生效服务合同的组合取值细则

### 4.12 合同主数据当前已冻结的关键原则
1. 合同属于主数据对象之一
2. 合同分为购销合同和服务补充合同 / 协议两类
3. 购销合同归属销售
4. 服务补充合同 / 协议归属服务部
5. 服务补充合同 / 协议分为维修、平台 / SIM、专项服务三类
6. 同一客户主体可存在多个合同
7. 设备保修期建议值按关联合同号走
8. 发货信息中的合同号可作为设备追溯合同的依据
9. 平台 / SIM 价格规则在系统中属于建议值，不是锁死规则
10. 若有当前有效服务补充合同 / 协议，则优先按其给平台 / SIM 建议值
11. 若无当前有效服务补充合同 / 协议，则按同一客户主体历史购销合同最低价给建议值
12. 合同冲突时，服务补充合同 / 协议优先只影响平台 / SIM 建议值，不覆盖原合同事实
13. 最终是否采用平台 / SIM 收费建议值，由相关业务员决定；该决定不影响协议是否当前有效、主数据事实或 SLA 考核口径
14. 服务补充合同 / 协议的当前有效判断、默认延续与关闭规则，以 CHAIN_25 为准
15. 服务时效是指响应、到场、完修等售后时效约定
16. 若当前有效服务补充合同 / 协议中存在服务时效约定，则维修工单优先引用该约定进行考核；若不存在，则回落公司默认标准
17. 服务时效定义及考核边界以 CHAIN_26 为准
18. 同一客户主体原则上不应存在约定项相同的多份当前有效服务补充合同 / 协议
19. 若现实中存在多份约定项相同的当前有效服务补充合同 / 协议，则取签订时间更晚的一份作为当前建议依据

## 5. 机型主数据

### 5.1 机型主数据定位
机型主数据用于承接“这一种机器是什么”的基础定义，是机器主数据、BOM、机型资料、运输机型分类等内容的上游对象。

机型主数据解决的不是“某一台机器是谁”，而是：
- 某一种机器如何被识别
- 该机型叫什么
- 该机型属于哪一类产品
- 该机型有哪些基础说明
- 该机型对应什么 BOM
- 该机型在物流计费上属于哪种运输机型分类

说明：
- 机型主数据不替代机器主数据
- 机型主数据也不替代 BOM
- 机型主数据的作用，是作为“机器类型”这一层的统一上游对象
- 机器主数据解决单台设备身份，机型主数据解决机器类型定义

### 5.2 机型唯一识别口径
当前冻结口径为：

1. 机型主数据的唯一识别口径为**产品型号**
2. 产品型号即机器型号、机型，代表一种机器类型
3. 产品型号用于唯一识别该机型
4. 产品名称用于展示和业务沟通
5. 当前阶段不以产品名称作为机型唯一识别口径

示例理解：
- 例如 `FVM-CP23B2NIC-FSBS-1`，代表一种具体机型
- 该类编码用于区分不同机器类型
- 产品名称可作为中文名称使用，但不替代产品型号的唯一识别作用

说明：
- 机器号是单台机器的身份证号
- 产品型号是机型主数据的识别口径
- 两者不是同一个层级

### 5.3 机型主数据的基础内容
当前阶段，机型主数据主要承接以下内容：

1. 产品型号
2. 产品名称
3. 财务机型
4. 运输机型
5. 机型分类
6. 机型小分类
7. 机型描述

说明：
- 产品型号用于识别机型
- 产品名称通常会与合同、销售沟通发生互动
- 财务机型当前只做记录
- 运输机型用于物流推荐计价
- 机型分类与小分类用于统计与归类
- 机型描述用于帮助销售和售后快速理解该机型配置与特点

### 5.4 机型与机器关系
当前冻结口径为：

1. 一个机型可对应多台机器
2. 一台机器只能对应一个机型
3. 机器号对应的机型不会变化
4. 机型是机器主数据的重要上游对象
5. 机型定义“机器是什么类型”，机器定义“具体是哪一台设备”

说明：
- 机型与机器的关系为一对多
- 机型解决类型层问题
- 机器解决单机身份层问题

### 5.5 机型与合同关系
当前冻结口径为：

1. 合同中可引用机型
2. 产品名称通常会与合同内容发生互动
3. 合同约定的是某机型的销售与交付条款
4. 合同不替代机型主数据本身
5. 机型主数据应独立存在，不依赖某一份合同才成立

说明：
- 合同会写“卖什么机型、卖多少台”
- 但机型本身是主数据对象，不是合同附属信息

### 5.6 机型与 BOM 关系
当前冻结口径为：

1. BOM 是机型视角数据
2. 一个机型对应一份当前有效 BOM
3. BOM 以机型为上游对象存在
4. BOM 用于描述该机型的组成结构
5. 机型主数据不替代 BOM 文件

说明：
- 机型定义“这是什么机型”
- BOM 定义“这个机型由哪些部位、哪些零部件组成”
- 两者相关联，但不是同一个对象

### 5.7 机型资料口径
当前冻结口径为：

1. 机型资料包括视频、PPT、说明资料等内容
2. 机型资料需要按机型聚合查看
3. 一个机型可以关联多份资料
4. 一份资料也可以关联多个机型
5. 当前阶段不要求机型资料与机型形成一对一关系

说明：
- 现实中多个机型差别很细微
- 因此销售和售后使用的机型资料、视频、PPT，可能是共用的
- 当前阶段允许多个机型共享同一份资料
- 但这种“资料共享”不影响机型主数据本身独立存在

### 5.8 机型资料与 BOM 的区分
当前冻结口径为：

1. 机型资料与 BOM 不是同一个对象
2. 机型资料可以跨多个机型共享
3. BOM 不应按资料共享逻辑处理
4. 每个机型仍应有自己对应的当前有效 BOM
5. 即使多个机型共用同一份资料，也不代表它们共用同一份 BOM

说明：
- 机型资料用于帮助理解、销售、培训、售后参考
- BOM 用于描述具体机型组成结构
- 因此资料可共享，BOM 不应直接共享

### 5.9 运输机型口径
当前冻结口径为：

1. 机型主数据中应维护运输机型分类
2. 当前运输机型分类为：
   - 小
   - 标准
   - 大
3. 运输机型分类属于预先归类
4. 运输机型分类用于物流计费推荐值计算
5. 运输机型分类不直接锁死最终财务入账金额

说明：
- 运输机型分类服务于物流合同推荐计价
- 当前阶段应作为机型主数据的一部分存在
- 最终物流入账金额仍允许管理员修正

### 5.10 机型分类口径
当前冻结口径为：

1. 机型需要有分类
2. 机型分类用于统计、归类与业务视角管理
3. 当前已知分类包括但不限于：
   - 饮料机
   - 综合机
   - 格子柜
   - 咖啡机
   - 粉料机
4. 每个大分类下还可以有小分类
5. 当前阶段先冻结“分类 + 小分类”的结构，不展开更细的字典规则

说明：
- 机型分类首先服务于业务理解和统计分析
- 不用于替代产品型号的唯一识别作用

### 5.11 财务机型口径
当前冻结口径为：

1. 机型主数据中可以记录财务机型
2. 当前阶段，财务机型只做记录
3. 当前版本不展开财务机型的详细应用规则
4. 当前版本不把财务机型作为机型唯一识别口径

说明：
- 财务机型当前只保留记录口径
- 后续是否用于更细财务统计或对账规则，留待后续专题展开

### 5.12 机型描述口径
当前冻结口径为：

1. 机型主数据中可以保留机型描述

- 三者作用不同

### 5.13 机型主数据当前边界
当前阶段只冻结以下内容：

## 6. 机器主数据

### 6.1 机器定位
机器主数据用于承接实体机器的唯一识别、型号归属、当前所在状态以及跨客户流转历史，是发货、运输异常、工单等链路的重要基础对象。

说明：
- 机器主数据不等于合同
- 机器主数据不等于客户
- 机器可关联合同、客户、发货、运输异常、工单等业务记录
- 机器主数据的核心作用，是唯一识别一台实体机器并保留其历史轨迹
- 本章中的“当前归属客户”默认指**客户业务单位**

### 6.2 机器唯一识别口径
当前冻结口径为：

1. 机器号就是机器编号
2. 机器号是机器的身份证号
3. 机器号全局唯一
4. 系统以机器号作为机器主数据的唯一识别口径

### 6.3 机器号生成口径
当前冻结口径为：

1. 机器号在**生产前确定**
2. 机器号通常由两部分组成：
   - **番号**
   - **编号**

示例：
- `LA26C205001`

可拆解为：
- `LA26C205`：番号
- `001`：编号

补充说明：
- `LA` 代表量产
- `26` 代表 2026 年
- `C` 代表 3 月
- `205` 代表 `26C` 下的某个制番
- `001` 代表该制番下的第一台机器

### 6.4 特殊机器号口径
当前冻结口径为：

1. 大多数机器号遵循“番号 + 编号”的规则
2. 个别特殊机型可能只输入纯数字
3. 特殊机型不强制遵循通用拆解规律
4. 系统不能把机器号拆解规则视为绝对固定规则

### 6.5 机器入系统口径
当前冻结口径为：

1. 机器在**发货准备前写入系统**
2. 写入系统后即进入机器主档
3. 因此机器在未发货前，也应先存在机器主数据

### 6.6 机器与型号关系
当前冻结口径为：

1. 一台机器只能对应一个机器型号
2. 机器号对应的型号不会变化
3. 机器型号是机器主数据的重要属性之一

### 6.7 机器号生命周期口径
当前冻结口径为：

1. 机器号一旦生成，终身不变
2. 即使机器返厂维修、跨客户流转、再次发货，仍保留原机器号
3. 机器报废后，机器号仍然保留
4. 报废机器号不复用

### 6.8 机器与合同关系
当前冻结口径为：

1. 机器可通过发货记录关联到某次合同
2. 发货信息中存在合同号，因此机器可追溯到对应合同
3. 机器不是永久绑定某个合同
4. 同一台机器在后续流转中，可关联不同业务记录

### 6.9 机器与客户关系
当前冻结口径为：

1. 机器可能在不同客户业务单位之间流转
2. 机器应保留跨客户业务单位流转历史
3. 机器当前归属只反映当前状态，不代表永久归属

当前口径下，机器当前归属可表现为：
- **归属客户**
- **在库**
- **在途**

补充说明：
- 已发货并签收的机器，可写归属客户业务单位
- 还未发货的机器，视为在库
- 已发货但未签收的机器，视为在途

### 6.10 机器与运输异常关系
当前冻结口径为：

1. 运输异常可按机器号发起
2. 运输异常中的原异常机器与补发承接机器可以不是同一台机器
3. 若运输异常中补发的是同型号新机，则补发机器必须使用新的机器号
4. 原受损机器修复后再次流转，仍保留原机器号，不新建机器主档

### 6.11 机器与工单关系
当前冻结口径为：

1. 工单应以机器号作为核心追踪对象之一
2. 机器维修、异常、历史处理情况都应可回到机器号维度查看

### 6.12 机器主数据当前边界
当前阶段只冻结以下内容：

1. 机器号是机器主数据的唯一识别口径
2. 机器号全局唯一
3. 机器号在生产前确定
4. 机器号通常可拆为番号和编号两部分
5. 个别特殊机型可能使用纯数字机器号
6. 机器在发货准备前写入系统并进入机器主档
7. 一台机器只能对应一个机器型号，且型号不变
8. 机器号终身不变，报废后仍保留且不复用
9. 机器可通过发货记录关联合同，但不永久绑定合同
10. 机器可在不同客户业务单位之间流转，并应保留历史轨迹
11. 机器当前归属可表现为归属客户、在库、在途
12. 运输异常补发新机必须使用新的机器号
13. 原机器返厂修复后再次流转，仍使用原机器号
14. 工单应以机器号作为核心追踪对象之一

当前阶段暂不展开：
- 机器号拆解的程序规则
- 机器状态流转细节
- 机器与仓库库存的详细联动方式

### 6.13 机器主数据当前已冻结的关键原则
1. 机器号就是机器编号
2. 机器号是机器的身份证号
3. 机器号全局唯一
4. 机器号在生产前确定
5. 机器号通常由番号和编号两部分组成
6. 个别特殊机型可能使用纯数字机器号
7. 机器在发货准备前进入系统并形成机器主档
8. 一台机器只能对应一个机器型号
9. 机器号对应的型号不会变化
10. 机器号一旦生成终身不变

## 7. 机器关键零部件 SN 主数据

### 7.1 定位
机器关键零部件 SN 主数据用于承接单台机器与关键零部件序列号（SN）的绑定关系，为保内维修确认、关键零部件更换、历史追溯、防止客户以旧换新等场景提供依据。

说明：
- 本章不替代机器主数据
- 本章不替代 BOM / 零部件主数据
- 本章也不替代工单记录
- 本章的核心作用，是在机器主档之外，补齐“这台机器当前关键零部件装的是哪个 SN、历史上换过哪些 SN”的统一口径
- 本章中的关键零部件，默认指需要重点做 SN 管理与保内核验的关键部品

### 7.2 与机器、BOM、零部件的关系
当前冻结口径为：

1. 关键零部件 SN 管理属于机器主数据的延伸
2. 机型决定“哪些部位属于关键零部件管理范围”
3. BOM / 零部件主数据承接的是机型组成结构与物料定义
4. 机器关键零部件 SN 主数据承接的是单台机器的实际绑定关系
5. BOM 解决“应装什么”，关键零部件 SN 主数据解决“实际装的是哪个 SN”

说明：
- BOM 是机型视角数据
- 零部件是物料视角数据
- 关键零部件 SN 绑定是单机实例视角数据
- 三者相关联，但不是同一个对象

### 7.3 机型关键件位口径
当前冻结口径为：

1. 不同机型的关键零部件范围可以不同
2. 关键零部件应按机型定义
3. 机型层应先定义“关键件位”
4. 关键件位可挂接到 BOM 中对应的部位 / 索引
5. 关键件位定义的是“这一类关键位置需要做 SN 管理”，而不是单台机器实例关系

说明：
- 当前阶段不直接把“关键零部件”理解成全局统一清单
- 更准确地说，是“某机型下哪些关键件位需要做 SN 管理”
- 这样更适合承接不同机型关键零部件不同的现实

### 7.4 单机当前绑定口径
当前冻结口径为：

1. 单台机器在每个关键件位上，任一时点只应有一个当前有效 SN
2. 当前有效 SN 表示该机器该关键件位当前实际绑定的关键零部件 SN
3. 当前有效 SN 属于机器主数据的一部分
4. 当前有效 SN 不等于工单现场填写值
5. 当前有效 SN 以统一上传并确认后的结果为准

说明：
- 当前绑定关系是主数据口径
- 工单现场记录只是业务事实，不直接等于主数据当前值

### 7.5 出厂绑定口径
当前冻结口径为：

1. 机器出厂时，关键零部件的 SN 可与机器号形成绑定关系
2. 出厂绑定关系属于机器关键零部件 SN 主数据的重要来源之一
3. 机器出厂时并不要求所有关键件位都已经在系统中完整存在
4. 因此关键零部件 SN 绑定允许后续统一补上传
5. 当前阶段不把“出厂时必须全部绑定完整”作为硬性前置条件

说明：
- 现实中关键零部件 SN 可能隔一段时间统一上传
- 因此主数据需要允许“先存在机器，再补关键零部件 SN 绑定”

### 7.6 保内维修核验口径
当前冻结口径为：

1. 本章建立的核心目的之一，是支撑保内设备涉及关键零部件维修时的核验
2. 服务经理到现场后，需要读取将要换下来的关键零部件实际 SN
3. 若系统中已有出厂 / 当前绑定记录，则应对现场实际 SN 与系统记录做比对
4. 该比对用于判断现场关键零部件是否与系统已知记录一致
5. 该比对的业务目的，是防止客户以旧换新或关键零部件被私下调换

说明：
- 当前阶段先冻结“存在比对动作”这一原则
- 比对不一致时的业务处理动作，在工单链中再展开

### 7.7 出厂记录缺失时的口径
当前冻结口径为：

1. 若工单发生时，系统中不存在该机器对应关键件位的出厂 / 当前绑定记录
2. 则服务经理允许记录现场看到的实际旧 SN
3. 该现场旧 SN 先作为工单事实存在
4. 现场旧 SN 不能直接进入主数据当前绑定
5. 等后续统一上传补齐主数据记录后，再进行对比与确认

说明：
- 当前阶段承认“工单发生时主数据可能尚未完整”的现实
- 因此需要允许现场先记录事实
- 但现场旧 SN 不能直接替代统一上传的主数据口径

### 7.8 新 SN 更换口径
当前冻结口径为：

1. 当关键零部件发生更换时，服务经理需要记录换上的新 SN
2. 新 SN 的现场记录属于维修事实的一部分
3. 新 SN 后续应作为该机器该关键件位的新绑定关系来源
4. 旧 SN 从该机器当前绑定中退出
5. 新 SN 进入该机器当前绑定

说明：
- 本章中的“换绑”是指机器与关键零部件 SN 的绑定关系发生切换
- 新 SN 替换旧 SN 后，当前绑定关系随之更新

### 7.9 生效时点口径
当前冻结口径为：

1. 关键零部件更换的业务生效时点，以工单完修为准
2. 工单完修前，现场记录的新旧 SN 先作为工单事实存在
3. 工单完修后，关键零部件 SN 绑定关系应按本次维修结果切换
4. 当前阶段先冻结“工单完修是换绑生效时点”这一原则
5. 当前版本不展开更复杂的中间状态

说明：
- 完修前，维修仍处于过程态
- 完修后，才视为关键零部件更换正式成立

### 7.10 统一上传与主数据确认口径
当前冻结口径为：

1. 主数据中的关键零部件当前绑定关系，仍以统一上传并确认后的结果为准
2. 服务经理现场填写的旧 SN 不能直接进入主数据
3. 服务经理现场填写的新 SN 不在仓库端做记录
4. 新 SN 现场记录可作为后续主数据更新来源之一
5. 最终主数据仍应通过统一上传与确认流程回收到统一口径

说明：
- 当前阶段先承认现场记录与主数据更新之间存在时间差
- 主数据口径仍以统一上传为主
- 工单现场值主要用于支撑现场核验和后续确认

### 7.11 旧 SN 脱钩与再利用口径
当前冻结口径为：

1. 旧 SN 在换绑后退出该机器当前绑定关系
2. 旧 SN 退出当前绑定，不等于删除
3. 旧 SN 需要保留历史关系
4. 旧 SN 允许后续再次装到其他机器
5. 因此关键零部件 SN 不是一次性消耗型标识，而是可循环流转的实例对象

说明：
- 当前阶段先冻结“旧 SN 允许循环使用”这一原则
- 这样更符合维修后旧件再利用的现实

### 7.12 历史留痕口径
当前冻结口径为：

1. 关键零部件 SN 绑定关系需要保留历史
2. 当前绑定与历史绑定必须区分
3. 历史至少应能够追溯：
   - 哪台机器
   - 哪个关键件位
   - 旧 SN
   - 新 SN
   - 变更来源
4. 当前阶段不展开历史展示方式
5. 但必须冻结“历史留痕存在”这一原则

说明：
- 若只保留当前绑定、不保留历史，后续将无法追溯出厂关系和维修更换轨迹
- 因此关键零部件 SN 管理不能只做覆盖式更新

### 7.13 与工单关系
当前冻结口径为：

1. 工单是关键零部件现场核验与更换事实的重要来源
2. 保内维修涉及关键零部件时，工单需要引用本章数据进行核验
3. 当主数据缺失时，工单允许先记录现场旧 SN 和新 SN
4. 工单记录不直接替代主数据当前绑定
5. 工单完修后，本次维修事实可推动主数据后续更新

说明：
- 工单负责承接现场事实
- 主数据负责承接统一绑定关系
- 两者分工不同，但需要联动

### 7.14 当前边界
当前阶段只冻结以下内容：

1. 机器关键零部件 SN 管理属于机器主数据的延伸
2. 关键零部件按机型定义，不同机型范围可以不同
3. 机型层先定义关键件位
4. 关键件位可挂接 BOM 部位 / 索引
5. 单台机器在每个关键件位上，任一时点只应有一个当前有效 SN
6. 出厂绑定关系是关键零部件 SN 主数据的重要来源之一
7. 出厂时不强制要求所有关键件位都已完整绑定
8. 保内维修涉及关键零部件时，需要做现场实际 SN 与系统记录的比对
9. 若系统中没有出厂 / 当前绑定记录，则允许服务经理先记录现场旧 SN
10. 现场旧 SN 只作为工单事实存在，不能直接进入主数据
11. 更换关键零部件时，服务经理需记录新 SN
12. 新 SN 现场记录不在仓库端做记录
13. 工单完修是关键零部件更换与换绑的生效时点
14. 主数据当前绑定关系仍以统一上传并确认后的结果为准
15. 旧 SN 换下后退出当前绑定，但必须保留历史
16. 旧 SN 允许后续再次装到其他机器
17. 当前绑定与历史绑定必须区分
18. 工单负责记录现场事实，主数据负责承接统一绑定关系

当前阶段暂不展开：
- 比对不一致时的详细处理流程
- 关键件更换后的审批与复核机制
- SN 照片 / 附件留存要求
- 统一上传的文件格式与导入流程
- 历史关系的页面展示方式
- 与仓库端的更细联动逻辑

### 7.15 当前已冻结的关键原则
1. 机器关键零部件 SN 管理的目的之一，是防止客户以旧换新
2. 关键零部件按机型定义，不同机型范围可以不同
3. 机型层定义的是关键件位，不是单台机器实例关系
4. 单台机器在每个关键件位上，任一时点只应有一个当前有效 SN
5. 当前有效 SN 属于主数据口径
6. 保内维修涉及关键零部件时，需要读取现场旧 SN 与系统记录做比对
7. 若系统已有记录，则做一致性比对
8. 若系统无记录，则允许服务经理先记录现场旧 SN
9. 现场旧 SN 不能直接进入主数据
10. 更换关键零部件时，服务经理需要记录新 SN
11. 新 SN 现场记录不在仓库端做记录
12. 工单完修是换绑生效时点
13. 主数据当前绑定仍以统一上传并确认后的结果为准
14. 旧 SN 换下后退出当前绑定，但不删除
15. 旧 SN 允许后续再次装到其他机器
16. 当前绑定关系与历史绑定关系必须区分
17. 工单负责承接现场事实，主数据负责承接统一绑定关系

## 8. SIM 主数据

### 8.1 SIM 定位
SIM 主数据用于承接系统内所有 SIM 卡的唯一识别、当前绑定关系、当前归属客户以及历史流转轨迹，是发货准备、平台 / SIM 收费、工单处理等链路的重要基础对象。

说明：
- SIM 主数据不等于平台主数据
- SIM 与平台被放在一起处理，主要是因为二者通常都按年缴费，便于客户统一理解与收费
- SIM 本质上是独立主数据对象
- 本章中的客户默认指**客户业务单位**

### 8.2 SIM 唯一识别口径
当前冻结口径为：

1. SIM 卡以 **ICCID** 作为唯一识别口径
2. ICCID 全局唯一
3. 系统以 ICCID 识别一张 SIM 卡

### 8.3 SIM 入系统口径
当前冻结口径为：

1. SIM 卡必须先进入系统主数据
2. 所有卡号只能从主数据中选择
3. 不允许业务过程中临时手工自编卡号

说明：
- SIM 卡是先有主数据，再进入后续业务使用
- ICCID 是主数据通用录入规则中的唯一例外

### 8.4 SIM 套餐口径
当前冻结口径为：

1. SIM 卡套餐当前只分两种
2. 后续套餐类型可能变化
3. 套餐详情不在主数据中展开
4. 套餐详细规则写在 SIM 收费模块

### 8.5 SIM 与机器关系
当前冻结口径为：

1. SIM 卡是插拔卡
2. 因此 SIM 与机器不是永久强绑定关系
3. 一台机器如果有卡，则同一时刻只对应一个 ICCID
4. 一张卡在同一时刻也只允许绑定一台机器
5. 若发生换卡或换机，需先解绑原关系，再建立新关系

### 8.6 SIM 与设备关系
当前冻结口径为：

1. ICCID 不一定对应设备号
2. 可能存在客户将卡插入非本司设备的情况
3. 因此 SIM 主数据不能要求必须绑定本司设备号后才成立

### 8.7 SIM 与客户关系
当前冻结口径为：

1. SIM 不永久强绑客户
2. 但 SIM 需要有**当前归属客户**
3. 即使未绑定本司设备，也应能够锁定到客户

当前已明确的典型来源包括：
1. 出厂设备自带卡，后续被换到其他设备上
2. 客户直接从服务部购买 SIM 卡

### 8.8 SIM 生命周期口径
当前冻结口径为：

1. ICCID 永久保留
2. ICCID 不复用
3. SIM 可停用，也可重新启用
4. 因此 SIM 主数据需要有状态管理
5. 当前阶段只冻结“存在状态管理”这一原则，不展开状态细节

### 8.9 SIM 主数据当前边界
当前阶段只冻结以下内容：

1. ICCID 是 SIM 的唯一识别口径
2. ICCID 全局唯一
3. SIM 卡必须先进入主数据
4. 所有卡号只能从主数据中选择，不能临时自编
5. SIM 套餐当前分两种，详细规则写在 SIM 收费模块
6. SIM 是插拔卡，不与机器永久强绑定
7. 一台机器同一时刻只对应一个 ICCID
8. 一张卡同一时刻只绑定一台机器
9. 换卡或换机时，需要先解绑原关系
10. ICCID 不一定对应本司设备号
11. 即使不对应本司设备，也应能锁定到客户
12. SIM 不永久强绑客户，但要有当前归属客户
13. ICCID 永久保留，不复用，可重新启用
14. SIM 主数据不等于平台主数据
15. SIM 的“当前归属客户”和“当前结算客户”一定相同

当前阶段暂不展开：
- 状态流转细节
- 套餐规则细节
- SIM 与平台的详细绑定规则
- SIM 与收费、结算的详细执行逻辑

### 8.10 SIM 主数据当前已冻结的关键原则
1. ICCID 是 SIM 的唯一识别口径
2. ICCID 全局唯一
3. SIM 卡必须先进入主数据
4. 所有卡号只能从主数据中选择
5. 不允许临时手工自编卡号
6. SIM 套餐当前只分两种
7. 套餐细节写在 SIM 收费模块
8. SIM 是插拔卡，不与任何主数据永久强绑定
9. 一台机器如果有卡，同一时刻只对应一个 ICCID
10. 一张卡同一时刻只允许绑定一台机器
11. 发生换卡或换机时，必须先解绑原关系
12. ICCID 不一定对应设备号
13. 客户可能将卡插到非本司设备中使用
14. SIM 不永久强绑客户，但需要有当前归属客户
15. 出厂带卡与服务部单独售卡两类场景都应能落到客户
16. ICCID 永久保留，不复用
17. SIM 可停用，也可重新启用
18. SIM 主数据不等于平台主数据
19. SIM 与平台被放在一起处理，主要因为二者通常都按年缴费
20. SIM 的“当前归属客户”和“当前结算客户”一定相同

## 9. 平台主数据

### 9.1 平台定位
平台主数据用于管理我司自有的售货机智能化平台相关基础资料，承接平台配置规则、客户平台账号、平台与机器关系以及平台费记录的基础依据。

说明：
- 平台主数据管理的是我司自有平台
- 平台主数据不等于机器主数据
- 平台主数据不等于客户主数据
- 平台费的详细计费规则不在本文展开，已写入平台费计费提醒与来源报价单生成链
- 本章中的客户默认指**客户业务单位**

### 9.2 平台数据分层口径
当前冻结口径为：

平台数据分为两层：

1. **平台配置层**
2. **客户平台层**

说明：
- 平台配置层用于定义不同平台本身的基础规则
- 客户平台层用于承接某客户在某个平台下的实际账号 / 实例

### 9.3 平台配置层
当前冻结口径为：

平台配置层当前管理内容包括：

1. 平台名
2. 平台代码
3. 查重规则
4. 机构代码 / 简称校验规则
5. 平台商应付计点规则
6. 平台商应付单价

### 9.4 平台配置规则口径
当前冻结口径为：

不同平台可配置不同的规则，例如：

1. 某些平台要求邮箱全局唯一
2. 某些平台要求电话全局唯一
3. 某些平台对机构代码有格式要求
4. 某些平台对简称有格式要求
5. 某些平台要求机构代码 / 简称必须满足特定规则，例如必须为大写字母
6. 不同平台可配置不同的平台商应付计点规则
7. 不同平台可配置不同的平台商应付单价

说明：
- 查重规则和机构代码 / 简称校验规则都属于平台配置层规则
- 平台商应付计点规则和平台商应付单价，也属于平台配置层规则
- 这些规则由平台配置定义，不由客户平台层单独决定
- 平台商应付计点规则不等于客户收费点数规则

### 9.5 客户平台层
当前冻结口径为：

客户平台层用于管理客户在具体平台下的实际平台账号 / 平台实例。

客户平台层的核心作用包括：

1. 承接客户在平台中的实际使用关系
2. 承接客户平台账号与机器的关系
3. 承接平台费记录的业务基础
4. 承接平台账号在客户之间流转的关系
5. 承接平台下机构与平台商应付计费关系的业务基础

### 9.6 客户平台与平台配置关系
当前冻结口径为：

1. 一个客户平台账号只能对应一个具体平台配置
2. 客户平台必须挂在某一个具体平台之下
3. 不存在脱离平台配置单独存在的客户平台账号

### 9.7 客户平台与客户关系
当前冻结口径为：

1. 同一客户业务单位可能存在多个平台账号
2. 同一客户业务单位在同一平台下，也可能存在多个平台账号
3. 同一客户业务单位也可能跨多个平台存在多个平台账号
4. 同一平台账号在同一时刻只能隶属于一个客户业务单位
5. 平台账号允许在客户之间换绑

说明：
- 客户之间发生设备交易时，平台账号可能随业务一起转移
- 因此平台账号不能视为永久绑定某一客户

### 9.8 客户平台下的唯一性口径
当前冻结口径为：

1. 机构代码 / 简称只在**同一平台下唯一**
2. 不要求跨所有平台全局唯一
3. 在要求机构代码 / 简称的平台下，同一平台下的多个平台账号都必须拥有各自不同的机构代码 / 简称
4. 平台唯一性判断必须结合具体平台配置来看

### 9.9 平台账号状态口径
当前冻结口径为：

1. 客户平台账号需要有状态栏
2. 当前阶段只冻结“存在状态管理”这一原则
3. 状态名称与状态流转规则后续再定

### 9.10 平台与机器关系
当前冻结口径为：

1. 平台账号可以关联多台机器
2. 平台账号与机器是多对一关系中的“一”端
3. 多批次、不同合同下的机器可以挂在同一平台账号下
4. 平台账号下的机器关系在出厂时即可写入

说明：
- 同一客户第一份合同下的机器与第二份合同下的机器，挂在同一平台账号下是常见场景
- 平台账号承接的是客户的平台使用关系，不按单次合同切断

### 9.11 平台与费用关系
当前冻结口径为：

1. 系统对平台的管理重点包括：
   - 平台主数据
   - 平台费记录
2. 平台主数据中的平台配置层，可以承接平台商应付计点规则与平台商应付单价
3. 客户收费侧的平台费计费规则，不在平台主数据中展开
4. 平台商应付的月度对账、账单、付款逻辑，不在平台主数据中展开
5. 平台费计费规则已写入平台费计费提醒与来源报价单生成链

### 9.12 平台主数据当前边界
当前阶段只冻结以下内容：

## 10. BOM / 零部件主数据

### 10.1 BOM / 零部件定位
BOM / 零部件主数据用于承接机型组成结构、爆炸图、零部件清单以及零部件公共资料，为工单、仓库、备件、报价、维修、查询等业务提供基础依据。

说明：
- BOM 是机型视角的数据
- 零部件是物料视角的数据
- 二者需要都可查询，但不是同一个对象
- 仓库与工单涉及的物料统一来源于零部件主数据范围，而不是仅来源于 BOM 清单

### 10.2 BOM 与机型关系
当前冻结口径为：

1. 每个机型都会有一份 BOM 文件
2. BOM 文件用于描述该机型的组成结构
3. 一个机型对应一份当前有效 BOM
4. BOM 以机型为上游对象存在

### 10.3 BOM 内容口径
当前冻结口径为：

BOM 文件包含两部分内容：

1. 每个部位的**爆炸图**
2. 对应的**零部件清单**

补充说明：
- 一个爆炸图对应一份零部件清单
- 爆炸图与清单是一一对应关系

### 10.4 BOM 分页口径
当前冻结口径为：

1. 每张爆炸图上的零部件从 **1** 开始编号
2. 该编号对应零部件清单中的**索引编号**
3. 因此 BOM 必须分页管理
4. 每一页对应一张爆炸图及其对应清单

### 10.5 零部件清单口径
当前冻结口径为：

每份零部件清单包含以下内容：

1. 索引编号
2. 部品名称
3. 可选件
4. 资材及规格
5. 图号及尺寸
6. 个数
7. 销售价格
8. 提供价格

补充说明：
- “图号及尺寸”是当前表头名称
- 但实际核心数据只有**图号**

### 10.6 零部件分类口径
当前冻结口径为：

零部件主数据应覆盖三类物料：

#### 10.6.1 BOM 零部件
- 出现在 BOM 爆炸图和零部件清单中
- 一般有图号
- 可被机型引用

#### 10.6.2 非 BOM 零部件
- 不出现在 BOM 中
- 但仍可归属某个或多个机型
- 可用于维修、替换、改装等业务场景

#### 10.6.3 消耗品
- 不归属于 BOM
- 一般没有图号
- 例如玻璃胶等
- 可用于仓库和工单领用

### 10.7 图号口径
当前冻结口径为：

1. 图号全局唯一
2. 图号代表一个零部件的唯一识别口径
3. 图号就是零部件的“身份证号”
4. 图号相同，代表同一种零部件
5. 零部件主数据中，有图号的零部件应以图号作为核心识别依据

### 10.8 无图号零部件口径
当前冻结口径为：

1. 部分不归属于 BOM 的零部件一般没有图号
2. 部分消耗品一般也没有图号
3. 系统可以为这类物料提供内部编码
4. 但业务上主要按**零部件名称 + 供应商**识别

说明：
- 无图号物料不应强行套用图号规则
- 这类物料仍属于零部件主数据范围

### 10.9 零部件与机型关系
当前冻结口径为：

1. 零部件主数据不隶属于某一个机型
2. 零部件是公共主数据
3. 不同机型可以引用同一个零部件
4. 不同机型存在大量通用零部件
5. 同一个图号可以被多个机型的 BOM 引用
6. 非 BOM 零部件也可能归属多个机型

### 10.10 可选件口径
当前冻结口径为：

1. 可选件是指该零部件可用于加装某个东西
2. 可选件口径本身与机型无关
3. 它反映的是零部件自身的可选属性，而不是特定机型下的临时说明

### 10.11 个数口径
当前冻结口径为：

1. 个数表示该零部件在对应机型中一共使用了多少个
2. 个数属于 BOM 引用关系下的使用数量
3. 个数不是零部件主数据本身的固定属性

### 10.12 价格口径
当前冻结口径为：

1. 销售价格是默认的对客户销售价格
2. 销售价格是报价单零部件价格的来源之一
3. 提供价格表示成本
4. 销售价格与提供价格都属于**零部件主数据默认价格**
5. BOM 只负责引用零部件，不单独定义默认价格来源

### 10.13 与仓库、工单的关系
当前冻结口径为：

1. 仓库所涉及的零部件来源于零部件主数据范围
2. 工单所涉及的零部件来源于零部件主数据范围
3. 该来源范围包括：
   - BOM 零部件
   - 非 BOM 零部件
   - 消耗品
4. 因此仓库与工单的物料来源不应只限制为 BOM 清单

### 10.14 查询口径
当前冻结口径为：

1. 系统需要支持 BOM 可查
2. 系统需要支持零部件可查
3. BOM 查询是机型视角查询
4. 零部件查询是物料视角查询

说明：
- BOM 查询用于查看某机型由哪些部位、哪些零部件组成
- 零部件查询用于查看某个零部件的资料、默认价格及其被哪些机型共用

### 10.15 BOM / 零部件主数据当前边界
当前阶段只冻结以下内容：

1. 每个机型都有一份 BOM 文件
2. BOM 文件包含爆炸图和对应零部件清单
3. 一个爆炸图对应一份零部件清单
4. BOM 必须分页管理
5. 每页爆炸图的编号与清单索引编号对应
6. 零部件清单包含索引编号、部品名称、可选件、资材及规格、图号及尺寸、个数、销售价格、提供价格
7. 零部件主数据应覆盖 BOM 零部件、非 BOM 零部件和消耗品
8. 图号全局唯一
9. 图号是有图号零部件的唯一识别口径
10. 图号相同就是同一种零部件
11. 零部件是公共主数据，不隶属于单一机型
12. 不同机型可共用同一零部件
13. 非 BOM 零部件也可能归属多个机型
14. 可选件表示该零部件可用于加装某个东西
15. 个数表示该零部件在对应机型中的使用数量
16. 销售价格和提供价格属于零部件主数据默认价格
17. 销售价格是报价单零部件价格来源之一
18. 仓库与工单涉及的物料来源于零部件主数据范围，而不是仅来源于 BOM
19. 除 ICCID 外，其他主数据允许先录入不存在的数据
20. 录入不存在的主数据后，应进入数据确认中心由管理员确认
21. 系统需要同时支持 BOM 查询和零部件查询

当前阶段暂不展开：
- BOM 版本管理细节
- 爆炸图文件上传与展示细节
- 零部件与库存、仓库、采购的详细联动方式
- 零部件价格变更历史规则
- 数据确认中心的详细处理流程

### 10.16 BOM / 零部件主数据当前已冻结的关键原则
1. BOM 与零部件属于两个相关但不同的主数据对象
2. BOM 是机型视角数据
3. 零部件是物料视角数据
4. 每个机型都有一份 BOM 文件
5. BOM 文件包含爆炸图和对应零部件清单
6. 一个爆炸图对应一份零部件清单

25. 系统需要做到 BOM 可查、零部件也可查

## 11. 机器现场信息主数据（客户联系人 / 工单地址 / 机器点位）

### 11.1 机器现场信息定位
机器现场信息主数据用于承接机器在客户现场长期会被重复引用的使用信息，为售后工单、再次上门、移机、巡检、培训、回访等业务提供统一基础资料。

本章中的机器现场信息，当前主要包括：
1. 机器当前现场联系人
2. 机器当前工单地址
3. 机器当前点位

说明：
- 本章不替代客户主数据
- 本章不替代机器主数据
- 本章也不替代工单记录
- 本章的核心作用，是在机器主档之外，补齐“这台机器当前在客户现场如何被找到、由谁配合、摆放在哪里”的统一口径
- 本章中的“客户”默认指**客户业务单位**

### 11.2 机器现场信息与业务单据的关系
当前冻结口径为：

1. 机器现场信息属于主数据范围
2. 工单中的联系人、地址、点位属于业务单据中的现场信息
3. 机器现场信息主数据用于沉淀机器当前有效现场事实
4. 工单不替代机器现场信息主数据
5. 机器现场信息主数据也不替代工单过程记录

说明：
- 工单负责记录一次事件中的现场情况
- 主数据负责保存当前有效值
- 历史记录负责承接变化轨迹

### 11.3 联系人口径
当前冻结口径为：

1. 本章中的联系人，默认指**机器当前现场联系人**
2. 该联系人可能只是客户现场的底层运营人员
3. 该联系人可能只负责 1 台或几台机器
4. 因此该联系人不要求先抽成完整的客户级联系人体系
5. 系统需要能够记录“这台机器当前应该联系谁”

说明：
- 本项目当前阶段不把这类联系人默认理解为客户高层联系人或固定商务联系人
- 这类联系人首先服务于工单受理、上门配合、再次联系、回访等业务场景
- 因此更适合作为机器维度的现场主数据来理解

### 11.4 工单地址口径
当前冻结口径为：

1. 工单地址用于描述机器所在现场的较粗地址
2. 工单地址通常不足以精确到区分同一现场内的多台机器
3. 工单地址主要用于说明机器大致所在地点
4. 工单地址与点位不是同一层信息
5. 工单地址不能替代点位

示例说明：
- 工单地址可写为：辽宁省大连市中山区大连医院
- 该地址可以帮助确认机器所在的大致现场
- 但不能直接区分同一医院内并排放置的多台机器

### 11.5 点位口径
当前冻结口径为：

1. 点位是售货机行业中用于描述**单台机器精确摆放位置**的业务概念
2. 点位通常比工单地址更精确
3. 点位用于区分同一现场内的不同机器
4. 本项目当前按“一机一点位”理解点位关系
5. 一台机器在任一时点，只应有一个当前点位

示例说明：
- A 机点位可写为：辽宁省大连市中山区大连医院三楼放射科门口左一
- B 机点位可写为：辽宁省大连市中山区大连医院三楼放射科门口左二

说明：
- 即使两台机器位于同一医院、同一科室门口，只要具体摆放位置不同，也应视为两个不同点位
- 因此点位不是泛地点概念，而是机器级精确位置概念

### 11.6 客户业务单位与机器现场信息关系
当前冻结口径为：

1. 一个客户下可有多个业务单位
2. 一个业务单位下可有多台机器
3. 每台机器应归属于一个当前客户业务单位
4. 每台机器同时应有自己的当前工单地址、当前点位、当前现场联系人
5. 机器现场信息是在客户业务单位基础上的进一步现场化表达

说明：
- 客户业务单位用于表达业务归属
- 工单地址用于表达大致现场
- 点位用于表达单机精确摆放位置
- 联系人用于表达当前现场对接人
- 四者不是同一层概念，不能互相替代

### 11.7 机器现场信息与工单关系
当前冻结口径为：

1. 工单中可以记录本次联系人
2. 工单中可以记录本次工单地址
3. 工单中可以记录本次点位
4. 工单是机器现场信息变化的重要发现来源
5. 但工单中的填写内容，不默认全部直接等同于主数据当前值

说明：
- 工单里的信息首先是“本次事件的现场信息”
- 是否沉淀为主数据当前值，需要作为当前有效事实来判断
- 因此工单与主数据之间应区分“本次记录”与“当前有效口径”

### 11.8 覆盖与历史留痕口径
当前冻结口径为：

1. 机器现场信息主数据需要保留当前值
2. 当联系人发生变化时，应更新机器当前现场联系人
3. 当工单地址发生变化时，应更新机器当前工单地址
4. 当点位发生变化时，应更新机器当前点位
5. 当前值更新时，需要保留变更历史
6. 无变化时，不应重复新增历史
7. 不应把每张工单都完整沉淀成一份新的永久主数据快照

说明：
- 本项目不建议采用“只覆盖、不留痕”的方式
- 本项目也不建议采用“每张工单都全量归档”的方式
- 更合理的口径是：当前值 + 变化留痕

### 11.9 机器现场信息主数据当前边界
当前阶段只冻结以下内容：

1. 机器现场信息属于主数据范围
2. 机器现场信息当前主要包括现场联系人、工单地址、点位
3. 联系人默认按机器当前现场联系人理解
4. 这类联系人可能只负责 1 台或几台机器，不要求先抽成完整客户级联系人体系
5. 工单地址用于描述较粗现场地址
6. 点位用于描述单台机器精确摆放位置
7. 点位通常比工单地址更精确
8. 本项目当前按一机一点位理解点位关系
9. 一台机器在任一时点只应有一个当前点位
10. 一台机器同时应有自己的当前工单地址、当前点位、当前现场联系人
11. 工单中的联系人、地址、点位是主数据变化的重要发现来源
12. 工单不替代机器现场信息主数据
13. 主数据保存当前值
14. 信息变化时需要保留历史
15. 无变化时不新增历史
16. 不按每张工单全量沉淀一份永久主数据快照

当前阶段暂不展开：
- 联系人的详细分类方式
- 一台机器是否允许存在多个同时有效联系人
- 工单地址与点位的标准化规则
- 从工单回写主数据的详细触发时机
- 历史记录的详细展示方式
- 与页面、字段、权限、报表的具体实现方式

### 11.10 机器现场信息主数据当前已冻结的关键原则
1. 机器现场信息属于主数据范围
2. 机器现场信息不等于工单记录
3. 本章中的联系人默认指机器当前现场联系人
4. 这类联系人可能只是客户现场底层运营人员
5. 这类联系人可能只负责 1 台或几台机器
6. 不要求先抽成完整客户级联系人体系
7. 工单地址用于描述机器所在现场的较粗地址
8. 工单地址不能替代点位
9. 点位是售货机行业中描述单台机器精确摆放位置的业务概念
10. 点位通常比工单地址更精确
11. 本项目当前按一机一点位理解点位关系
12. 一台机器在任一时点只应有一个当前点位
13. 一个客户下可有多个业务单位
14. 一个业务单位下可有多台机器
15. 每台机器应归属于一个当前客户业务单位
16. 每台机器同时应有自己的当前工单地址、当前点位、当前现场联系人
17. 工单中的联系人、地址、点位是机器现场信息变化的重要发现来源
18. 工单记录的是本次事件现场信息
19. 主数据保存的是机器当前有效现场信息
20. 当前值变化时需要保留历史
21. 无变化时不新增历史
22. 不应把每张工单都完整沉淀成新的永久主数据快照
