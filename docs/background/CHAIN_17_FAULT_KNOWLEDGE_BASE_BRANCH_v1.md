## CHAIN_17_FAULT_KNOWLEDGE_BASE_BRANCH_v1.md

## 1. 链条定位
故障库不是售后工单主链本身，而是工单侧的一条知识沉淀支链。

售后工单主链解决的是：
报修 → 受理 → 派单 → 处理 → 完修 → 回访 → 关闭

故障库解决的是：
把已经发生过的工单经验，按机型与故障主题进行归类沉淀，形成后续可复用的故障知识。

说明：
- 工单主链的目标，是完成一次服务处理
- 故障库的目标，是把处理经验沉淀下来，供后续类似问题参考
- 因此，故障库依赖工单，但不替代工单
- 故障库属于工单侧支链，不属于独立主交易流程

---

## 2. 本链条的业务目的
建立故障库的主要目的，不是单纯保存历史工单，而是把分散在工单里的处理经验整理成可复用知识。

当前主要目标包括：

1. 按机型沉淀常见故障经验
2. 把相似故障现象归为同一类故障主题
3. 记录该类故障最后是如何解决的
4. 记录服务经理的处理思路
5. 为其他遇到类似问题的人员提供参考思路
6. 为后续统计故障频率提供整理基础

说明：
- 故障库首先服务于经验复用
- 故障频率统计是后续附加价值
- 当前阶段，不把故障库定义成自动诊断系统
- 当前阶段，也不把故障库定义成单纯的工单查询列表

---

## 3. 故障库与工单主链的关系
故障库的数据来源于工单，但工单与故障库不是同一个对象。

当前冻结口径为：

1. 工单负责记录一次真实服务事件
2. 故障库负责沉淀可复用的故障经验
3. 工单中的自然语言故障现象、处理过程、换件情况、最终结果，都可以成为故障库的来源
4. 不是所有工单信息都原样等于故障库内容
5. 故障库是在工单基础上的归类整理结果

说明：
- 工单是原始事实
- 故障库是归类后的知识结果
- 因此，故障库不是对工单的简单复制

---

## 4. 故障库的组织方式
当前建议按以下结构组织故障库：

机型 → 故障主题 → 案例 / 解决经验

说明：
- 机型是上层分类入口
- 故障主题用于归并相似故障
- 案例用于承接真实工单中的处理经验
- 解决经验用于沉淀最终可复用内容

这样组织的目的，是让用户先从机型进入，再快速定位该机型下常见故障主题，而不是在大量工单中逐条翻找。

---

## 5. 按机型分类的口径
故障库当前按机型分类。

当前冻结口径为：

1. 每条故障知识都应归属于某个机型
2. 同一机型下可以有多个故障主题
3. 同一故障主题下可以沉淀多个真实案例
4. 不同机型即使故障现象相似，也不默认视为同一条故障知识
5. 后续查看故障知识时，应优先从机型视角进入

说明：
- 当前阶段先按机型分类，便于经验复用
- 不先按客户、机器号、服务经理作为一级分类
- 机器号仍保留在工单层用于追溯，但不是故障库的一级入口

---

## 6. 故障主题的口径
故障主题，是把相似故障现象归并后的统一知识主题。

当前冻结口径为：

1. 工单中的故障现象通常是自然语言描述
2. 相同故障在不同工单中，表述方式可能不同
3. 因此需要把不同说法归并为同一故障主题
4. 故障主题不是工单原文，而是归类后的统一主题
5. 故障库最终对外复用的核心入口，应是故障主题，而不是工单原文标题

示例理解：
- 工单里可能出现“屏幕不亮”“开机黑屏”“显示没反应”等不同说法
- 这些说法在业务上可能被归并为同一故障主题
- 故障主题下面再沉淀不同案例和最终处理方式

说明：
- 当前阶段不要求故障主题一开始就非常细
- 重点是先形成可复用的归类框架

---

## 7. 案例与解决经验的口径
故障库不是只保存一个故障主题名称，还需要保存该主题下的案例与经验。

当前建议沉淀的内容包括：

1. 常见故障现象
2. 常见排查思路
3. 常见处理过程
4. 最终解决办法
5. 服务经理的处理思路
6. 涉及的典型零部件
7. 相关工单来源

说明：
- 故障主题用于统一归类
- 案例用于支撑该主题的真实来源
- 解决经验用于后续复用
- 因此，故障库既不是只有标签，也不是只有工单原文，而是“主题 + 案例 + 经验”的组合

---

## 8. 自然语言与智能归类口径
当前已知业务现实是：工单中的故障现象通常为自然语言描述，因此单纯依靠固定标签无法覆盖全部情况。

当前冻结口径为：

1. 工单中的故障现象允许继续使用自然语言描述
2. 故障库需要对相似故障进行智能分析与相似归类建议
3. 智能分析的作用，是辅助归类，不是直接替代人工判断
4. 当前阶段不把系统定义为自动最终裁决者
5. 当前阶段，系统更适合作为“相似故障建议工具”

说明：
- 同类故障在不同工单中的说法可能不同
- 因此需要引入智能归类思路
- 但当前阶段不能把故障库完全建立在自动判定之上

---

## 9. 归类确认口径
故障主题的归类，需要有人进行确认。

当前冻结口径为：

1. 当前由客服承担归类确认工作
2. 该确认主要用于形成故障库的归档结果
3. 当前阶段保留技术校正空间
4. 后续如有需要，可由服务经理或技术负责人对归类结果进行校正
5. 当前版本先不展开更复杂的多角色确认机制

说明：
- 当前先由客服承担归类确认，有利于形成基础整理机制
- 但考虑到故障归类带有技术判断属性，因此当前不把客服永久写死为唯一技术确认角色
- 当前版本只冻结“客服先承担确认工作，并保留技术校正空间”的口径

---

## 10. 与机型资料的关系
故障库需要与机型资料放在一起看，但两者不是同一个对象。

当前冻结口径为：

1. 机型资料包括视频、PPT等资料
2. 故障库需要和机型资料一起按机型聚合查看
3. 机型资料与故障库共享同一个机型视角
4. 但机型资料不等于故障库
5. 故障库也不等于机型主数据本体

说明：
- 机型资料解决“了解这个机型”
- 故障库解决“这个机型常见故障怎么处理”
- 两者应在同一机型视角下联动，但不能混成同一对象

---

## 11. 故障库的使用价值
故障库建立后，主要服务于以下场景：

1. 客服受理时，快速了解该机型常见问题
2. 服务经理处理时，参考类似故障的排查思路
3. 新人员学习时，查看典型案例与处理方式
4. 后续统计某机型常见故障频率
5. 为后续质量分析、问题归纳、经验复盘打基础

说明：
- 当前阶段首先强调经验复用价值
- 统计分析价值为下一步扩展基础
- 当前阶段不把故障库直接定义成质量预警系统

---

## 12. 当前边界
当前阶段，本链条只冻结“故障知识沉淀”本身，不展开以下内容：

1. 批量不良预警的具体规则
2. 连续触发、时间窗口、阈值等预警算法
3. 页面、字段、按钮、报表实现方式
4. AI模型、向量检索、相似度算法等实现细节
5. 多角色审批或复杂权限机制
6. 与其他部门的联动流程

说明：
- 当前先把故障库本身的业务边界定义清楚
- 批量不良预警后续应单独作为另一条支链规则继续展开
- 当前版本不把两者混在一起

---

## 13. 当前冻结结论
当前可冻结的总口径为：

- 故障库属于工单侧的一条知识沉淀支链
- 故障库不替代工单主链
- 工单是原始事实来源，故障库是归类后的知识结果
- 故障库当前按机型分类
- 故障库的组织方式为：机型 → 故障主题 → 案例 / 解决经验
- 工单中的故障现象允许继续使用自然语言描述
- 相同故障在不同工单中可能存在不同表达方式
- 因此需要对相似故障进行智能分析与相似归类建议
- 当前阶段，智能分析只作为归类辅助，不作为最终裁决
- 当前由客服承担归类确认工作
- 当前同时保留技术校正空间
- 故障库需要沉淀最终解决办法
- 故障库需要沉淀服务经理的处理思路
- 故障库需要沉淀相关案例来源
- 故障库可沉淀涉及的典型零部件信息
- 故障库需要与机型资料一起按机型聚合查看
- 机型资料包括视频、PPT等内容
- 机型资料与故障库共享机型视角，但不是同一个对象
- 故障库当前主要服务于经验复用、知识整理和后续频率统计
- 当前阶段暂不展开批量不良预警规则
- 当前阶段暂不展开页面、字段、AI实现细节与复杂权限机制
