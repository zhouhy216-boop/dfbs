## 2. 报价单体系定位

### 2.1 报价单体系的作用
报价单体系用于承接各类对客收费需求。

统一商业逻辑为：

客户有需求
→ 我方整理需求
→ 不同承接角色分别填写自己负责部分的报价
→ 如有需要，由服务经理补充服务执行成本
→ 汇总形成完整报价单
→ 财务第一次确认并判断整张报价单是否亏钱
→ 若亏钱则进入领导审核；不亏钱时财务也可发起领导审核
→ 服务经理与客户确认报价
→ 转入对应业务链执行
→ 业务完成后财务第二次确认
→ 进入结算层处理回款 / 开票
→ 结算完成后整体完结

因此：
- 报价单是各类对客收费需求的统一承接对象
- 报价单处于各类收费需求的下游
- 不同来源的收费需求，最终统一进入报价单体系处理

### 2.2 报价单与结算的关系
当前冻结口径为：

- 报价单与结算是两层内容，不是同一层内容
- 报价单负责确定“该收什么钱”
- 结算负责处理“这些钱怎么收、怎么开票”

也就是说：

#### 报价单层负责
- 承接收费需求
- 确认收费内容
- 形成报价明细
- 确认最终应收内容
- 作为后续结算处理基础

#### 结算层负责
- 承接回款处理
- 承接开票处理
- 跟踪未回款 / 未开票金额
- 在结算完成后形成整体完结结果

### 2.3 普通报价单结算与合并结算的关系
当前冻结口径为：

- 普通报价单结算与合并结算，属于同一结算层规则
- 两者底层处理口径应保持一致
- 区别只在于：
  - 普通报价单结算：只处理单张报价单
  - 合并结算：处理多张报价单

报价单合并细则后续单独冻结，
本文只先冻结“报价单层”和“结算层”的边界。

## 3. 当前适用范围
这套流程当前只服务“对客收费场景”，免费需求不进入本报价单体系。

说明：
- 平台或SIM存在免费场景时，可在对应收费入口链中承接免费原因或免费依据
- 但免费平台、免费SIM不生成收费来源报价单
- 免费场景不进入对客收费需求单与报价单流程

当前已明确会进入这套流程的收费类型，包括：

1. 平台续费
2. SIM卡续费
3. 客户委托运输
4. 收费维修
5. 配件购买
6. 其他付费服务
   - 付费培训
   - 付费改造
   - 付费检修 / 巡检

补充说明：
- 当前有效服务补充合同 / 协议，对报价链的影响，当前只先冻结到“维修相关报价明细”的建议依据
- 这里的“维修相关报价明细”，包括维修服务类收费明细，以及维修过程中产生的配件类收费明细
- 若当前有效服务补充合同 / 协议中存在服务价格、配件价格等约定，则后续维修相关报价可据此作为建议依据
- 平台 / SIM 的建议值口径，仍以对应收费入口链及主数据口径为准，不在本文中重复展开
- 运输类收费当前不引用服务补充合同 / 协议作为建议依据

## 4. 需求单定位

### 4.1 需求单的作用
需求单用于承接经过人工确认后的正式收费需求内容。

需求单不是记录客户原始口头诉求，
而是经过业务人员整理、确认后的正式需求。

### 4.2 需求单与报价单的关系
- 需求单在前
- 报价单在后
- 报价是需求的结果，不是独立起点

### 4.3 需求单范围
- 当前需求单只服务收费场景
- 免费场景不并入
- 免费平台、免费SIM如需留痕，应在对应收费入口链中承接免费原因或免费依据，不转入报价单体系

### 4.4 需求单类型规则
当前冻结口径为：

- 一个需求单允许包含多个需求项
- 不同需求项可以属于不同需求类型
- 不同类型的需求项允许出现在同一张需求单中

例如：
- 一个需求单里可以同时包含上门、人工、配件
- 一个需求单里也可以同时包含维修服务与相关配件

若不同需求项需要不同管理员报价，
则由不同管理员各自填写自己负责的部分。

### 4.5 不同需求类型的表单
当前冻结口径为：

- 需求单可以包含不同类型的需求项
- 不同类型的需求项仍按各自类型要求填写
- 不同管理员只填写自己负责的需求项部分
- 当前阶段不展开字段细节

## 5. 报价单明细定位

### 5.1 报价单阶段本身就是独立明细
当前冻结口径为：

- 报价单不是只有总金额
- 报价单阶段本身就是独立明细
- 每条报价明细具备独立结算含义

例如：
- 上门费
- 单价 80
- 次数 2
- 金额 160

### 5.2 报价明细是后续结算基础
当前冻结口径为：

- 报价明细是后续回款和开票的处理基础
- 回款和开票都不只按报价单头处理
- 而应落到报价明细层处理

### 5.3 若预计需要拆分，应在报价单阶段先拆细
当前冻结口径为：

- 若某项费用未来需要独立回款
- 或需要独立开票
- 或结算口径不同

则应在报价单阶段先拆成独立明细。

### 5.4 维修相关报价明细的建议依据
当前冻结口径为：

- 当前有效服务补充合同 / 协议，可作为维修相关报价明细的建议依据
- 这里只先冻结到两类建议依据：
  1. 服务价格建议依据
  2. 配件价格建议依据
- 该建议依据只适用于维修相关报价明细
- 不适用于运输类收费
- 不替代平台 / SIM 的建议值口径

说明：
- 这里的“建议依据”，不是自动带价，不是强制覆盖，也不是锁死规则
- 最终报价仍由对应承接角色按原有报价流程填写
- 报价仍需经过财务第一次确认、服务经理对客确认，并按原流程转入后续业务链
- 若不存在当前有效服务补充合同 / 协议，或虽存在但未约定服务价格 / 配件价格，则仍按原有报价流程人工报价

## 6. 角色分工

### 6.1 服务经理
- 可能最先接到客户需求
- 可以发起需求单
- 是最终与客户确认报价的人
- 业绩归属到服务经理
- 如果需求发生变化，由服务经理发起需求修改
- 在结算层中，也负责提交回款记录或发起相关结算处理

### 6.2 客服代表
- 也可能最先接到客户需求
- 可以直接发起需求单
- 若不是自己负责后续，需要在单中选择对应服务经理
- 负责审核、补充、修改、整理需求内容
- 某些场景下，报价通过后，会转给客服代表去发起后续业务链
- 在结算层中，也可按职责参与相关结算处理

### 6.3 相关管理员
例如：
- 平台管理员
- 物流管理员
- 其他专业管理员

职责：
- 根据需求单确认自己负责部分的需求
- 填写自己负责部分的报价
- 他们也可以从中间节点直接发起报价
- 但仍需选择对应服务经理

说明：
- 当一个需求单中包含多种需求项时
- 不同管理员可分别填写自己负责的报价部分
- 若某需求项需要预估服务执行成本，则由服务经理作为该部分负责人填写
- 服务执行成本不是所有报价单都必填，只在需要服务经理预估执行费用的收费项中填写
- 最终共同形成同一张报价单中的完整报价内容

### 6.4 财务角色
财务存在两次与报价内容相关的确认：

#### 第一次财务确认
- 根据需求内容确认报价是否正确
- 核对价格、内容、结算口径等
- 基于整张报价单判断是否亏钱
- 若整张报价单亏钱，则自动进入领导审核
- 即使不亏钱，财务也可以主动发起领导审核
- 通过后，报价单才具备与客户确认并转入后续业务链的条件

#### 第二次财务确认
- 在业务执行完成后进行
- 用于确认最终服务内容与结算一致
- 第二次财务确认完成后，报价单具备进入结算层处理的条件

说明：
- 第二次财务确认，确认的是报价内容与实际服务是否一致
- 不是回款确认，也不是开票确认
- 回款 / 开票属于后续结算层处理

## 7. 当前主流程

### 7.1 基本流程
客户提出收费需求
→ 服务经理 / 客服代表 / 相关管理员，从对应节点发起需求单
→ 客服代表审核、补充、修改需求内容
→ 转给对应需求承接角色 / 相关管理员
→ 不同承接角色 / 管理员分别填写自己负责部分的报价
→ 如有需要，服务经理填写服务执行成本
→ 汇总形成完整报价单
→ 财务第一次确认并判断整张报价单是否亏钱
→ 若亏钱则进入领导审核；不亏钱时财务也可发起领导审核
→ 服务经理与客户确认报价
→ 按不同报价明细归属，转给对应负责人进入后续业务链
→ 业务执行
→ 如执行中需求变化，服务经理发起需求修改
→ 修改后重新走流程
→ 业务执行完成
→ 财务第二次确认
→ 报价单进入结算层
→ 在结算层处理回款 / 开票
→ 结算完成后整体完结

### 7.2 从中间节点发起
需求单 / 报价单不一定只能从最前面节点发起。

当前允许：
- 服务经理直接发起需求单
- 客服代表直接发起需求单（但要选择服务经理）
- 相关管理员从中间节点直接发起报价（但要选择服务经理）

因此，这套流程允许按实际业务接触点从中间节点进入，
类似平台申请流程。

## 8. 第一次财务确认后的转业务链逻辑
财务第一次确认
→ 若整张报价单亏钱，则自动进入领导审核
→ 若不亏钱，财务也可主动发起领导审核
→ 内部审核通过后，由服务经理与客户确认报价
→ 再按不同报价明细归属，转给对应负责人发起后续业务链

例如：
- 维修相关报价明细通过后，转客服代表，由客服代表发起工单
- 运输相关报价明细通过后，转物流相关负责人，进入物流执行链
- 平台 / SIM相关报价明细通过后，转平台管理员，进入开通 / 续费链
- 配件相关报价明细通过后，转仓库 / 配件相关负责人，进入发货链

补充说明：
- 若维修相关报价明细在报价阶段引用了当前有效服务补充合同 / 协议中的服务价格、配件价格约定，该引用只影响报价形成时的建议依据
- 报价通过后，后续业务执行仍按原有工单、物流、平台 / SIM、发货等业务链分别承接
- 报价链不因为引用了服务补充合同 / 协议，就替代下游业务链自身的执行规则

因此：
- 报价单只负责统一承接收费需求
- 真正的业务执行仍然回到各自业务链中完成
- 当一张报价单中包含多类报价明细时，不同明细可分别进入各自业务链

## 9. 报价修改逻辑

### 9.1 修改由谁发起
- 只能由服务经理发起

原因：
- 服务经理直接面对客户
- 最清楚客户需求变化
- 不能由其他角色直接改报价内容

### 9.2 修改的对象
- 修改的是需求
- 不是直接改报价

报价只是需求的结果。

### 9.3 修改后的处理方式
- 只要需求变更，就重新走一遍完整流程
- 即重新经过：
  - 对应承接角色 / 管理员重新填写自己负责部分的报价
  - 如有需要，服务经理重新填写服务执行成本
  - 财务第一次重新确认，并重新判断整张报价单是否亏钱
  - 若亏钱则重新进入领导审核；不亏钱时财务也可再次发起领导审核
  - 服务经理重新与客户确认报价

说明：
- 当前阶段不区分“只改一部分”还是“改整张单”
- 只要需求发生变化，仍按整张报价流程重走

### 9.4 改的是哪张报价单
- 改的是原报价单
- 不新开补充报价单

### 9.5 业务开始后是否允许改
- 允许
- 因为像维修这类业务，执行中实际内容可能发生变化

### 9.6 已回款 / 已开票后是否允许改
- 允许发生
- 如果回款金额未超过报价单金额，正常继续处理
- 如果涉及退款 / 退票，则需要走特殊流程 / 领导审核流程

### 9.7 下游业务链是否跟着重建
- 下游链条不变
- 已下发的任务允许与最终实际完成的任务不同
- 报价最终挂钩的是“最终实际提供了什么服务”

### 9.8 变更轨迹
- 原报价单允许变更
- 但必须保留变更轨迹

## 10. 报价单与结算层的边界

### 10.1 报价单层冻结到哪里
当前冻结口径为：

- 报价单层负责承接需求、形成报价、确认最终服务与金额
- 到业务完成并完成财务第二次确认时，报价内容层已经稳定
- 此后进入结算层处理

### 10.2 结算层承接什么
当前冻结口径为：

- 结算层承接回款
- 结算层承接开票
- 结算层按报价明细处理结算动作
- 普通报价单结算与合并结算属于同一结算口径

### 10.3 本文不展开结算细则
虽然本文已冻结报价单与结算的边界，
但以下内容仍留待后续单独讨论：
- 普通报价单如何回款
- 普通报价单如何开票
- 合并结算如何回款
- 合并结算如何开票
- 退款 / 退票 / 冲销细则

---

## 11. 报价单完结逻辑
当前冻结口径为：

- 报价单不会在第一次财务确认后就完结
- 第一次财务确认后，报价单具备转业务链执行的条件
- 在业务执行期间，如需求发生变化，允许服务经理发起修改
- 业务执行完成后，由财务第二次确认“最终服务内容与结算一致”
- 第二次财务确认完成后，报价单进入结算层处理
- 整体流程以结算完成作为完结标志

说明：
- 财务第二次确认，不等于整个收费流程完结
- 它表示报价内容层完成，并进入结算层
- 回款 / 开票完成后，整体才算最终完结

---

## 12. 当前已冻结的关键原则
1. 报价单体系只服务对客收费场景，免费需求不进入报价单体系；免费平台、免费SIM仅在各自入口链承接免费原因或免费依据
2. 报价单位于各类收费需求下游
3. 需求单在前，报价单在后
4. 一个需求单允许包含多个需求项
5. 不同需求项可以属于不同需求类型
6. 不同类型的需求项允许出现在同一张需求单中
7. 若不同需求项需要不同管理员报价，则由不同管理员各自填写自己负责的部分
8. 服务经理、客服代表、相关管理员都可能从不同节点发起
9. 财务第一次确认发生在服务经理与客户确认报价之前
10. 第一次财务确认除核对报价正确性外，还负责判断整张报价单是否亏钱
   - 若整张报价单亏钱，则自动进入领导审核
   - 即使不亏钱，财务也可以主动发起领导审核
   - 内部审核完成后，再由服务经理与客户确认报价
   - 第二次确认最终服务内容与结算一致
11. 完成对客确认后，才按不同报价明细归属转给对应负责人去发起后续业务链
12. 需求变更只能由服务经理发起
13. 服务经理改的是需求，不是直接改报价
14. 需求一旦变更，就重新走一遍完整流程
15. 改的是原报价单，不另开补充报价单
16. 业务执行期间允许修改
17. 下游业务链不因报价修改而重建
18. 原报价单必须保留变更轨迹
19. 报价单与结算是两层内容，不是同一层内容
20. 报价单负责确定“该收什么钱”，结算负责处理“这些钱怎么收、怎么开票”
21. 普通报价单结算与合并结算，底层规则应保持一致
22. 普通报价单结算与合并结算的区别，只在于结算范围是一张还是多张报价单
23. 报价单阶段本身就是独立明细
24. 报价明细是后续结算处理基础
25. 若预计需要独立回款或独立开票，应在报价单阶段先拆成独立明细
26. 当前有效服务补充合同 / 协议，对报价链的影响，当前只先冻结到维修相关报价明细的建议依据
27. 该建议依据只先包括服务价格建议依据和配件价格建议依据
28. 该建议依据不是自动带价，不是强制覆盖，也不是锁死规则
29. 若不存在当前有效服务补充合同 / 协议，或虽存在但未约定服务价格 / 配件价格，则仍按原有报价流程人工报价
30. 平台 / SIM 的建议值口径，仍以对应收费入口链及主数据口径为准，不在本文中重复展开
31. 运输类收费当前不引用服务补充合同 / 协议作为建议依据
32. 第二次财务确认完成后，报价单具备进入结算层处理的条件
33. 整体流程以结算完成作为完结标志
