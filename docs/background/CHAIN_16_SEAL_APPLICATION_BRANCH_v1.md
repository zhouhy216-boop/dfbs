## 2. 支线定位
盖章申请是一条独立支线，用于承接文件的盖章申请、审核和历史留痕。

当前冻结口径为：
1. 盖章只是一个申请入口
2. 盖章申请不代表合同 / 协议 / 报价单生效
3. 盖章申请不替代合同主数据、协议主数据、报价主流程
4. 盖章申请的主要目的，是避免重复录入，并形成统一留痕入口

---

## 3. 盖章申请适用对象
当前冻结口径为：

盖章申请当前适用于以下对象：

1. 服务合同 / 协议
2. 报价单
3. 其他文件

说明：
- 购销合同当前不纳入盖章申请支线
- 其他文件允许发起盖章申请，但只保留历史，不进入主数据

---

## 4. 服务合同 / 协议与盖章申请的关系
当前冻结口径为：

1. 服务合同 / 协议可以发起盖章申请
2. 盖章申请审核并完成后，不代表该合同 / 协议已经生效
3. 盖章完成后，也不代表该合同 / 协议已经自动成为下游业务可引用的当前有效依据
4. 盖章完成后，是否进入主数据、何时进入主数据，仍由管理员控制
5. 在进入主数据之前，该合同 / 协议不能作为正式主数据被其他业务引用
6. 即使已经进入主数据，是否构成当前有效服务补充合同 / 协议，仍应以 CHAIN_25 的规则判断
7. 下游业务是否可以引用该合同 / 协议，也应以各自主业务链规则为准，而不是以盖章状态为准

说明：
- 盖章完成只是形成了盖章申请结果
- 进入主数据，是后续独立动作
- 当前有效判断，是再下一层独立判断
- 因此，“已盖章”“已进入主数据”“当前有效可引用”不是同一件事
---

## 5. 报价单与盖章申请的关系
当前冻结口径为：

1. 报价单在生成后，可随时申请盖章
2. 报价单申请盖章，不代表报价单生效
3. 报价单申请盖章，不影响报价主流程状态
4. 盖章申请的作用，仅是让受理人知道对应的是哪一张报价单，并形成留痕

说明：
- 报价单盖章不改变报价确认、财务确认、领导审核等主流程口径
- 报价单盖章只是一条附属支线

---

## 6. 其他文件与盖章申请的关系
当前冻结口径为：

1. 其他文件允许发起盖章申请
2. 其他文件盖章申请完成后，只保留在盖章申请历史中
3. 其他文件不进入主数据
4. 其他文件不影响合同、协议、报价单等主业务对象状态

---

## 7. 进入主数据的边界
当前冻结口径为：

1. 不是所有经过盖章申请的文件都会进入主数据
2. 当前只有服务合同 / 协议存在“盖章完成后可由管理员选择何时进入主数据”的口径
3. 报价单不因盖章申请而进入主数据
4. 其他文件不因盖章申请而进入主数据
5. 服务合同 / 协议进入主数据，只代表其成为可被管理的主数据对象
6. 服务合同 / 协议进入主数据后，仍不等于已经构成当前有效依据
7. 服务补充合同 / 协议是否属于当前有效、是否可作为下游建议依据或考核依据，仍以 CHAIN_25 及相关下游业务链规则为准

说明：
- “进入主数据”解决的是对象承接问题
- “当前有效”解决的是业务引用资格问题
- 这两层边界必须分开

---

## 8. 生效边界
当前冻结口径为：

1. 盖章申请不代表文件生效
2. 文件是否生效，仍应以各自主业务链的规则为准
3. 因此：
   - 合同 / 协议是否生效，不由盖章申请决定
   - 报价单是否生效，不由盖章申请决定
   - 其他文件也不因盖章而自动获得业务效力
4. 对于服务补充合同 / 协议而言，盖章完成不代表其已经成为当前有效协议
5. 对于服务补充合同 / 协议而言，是否可被报价链、主数据建议值链、SLA考核链等下游业务引用，仍应分别按对应业务链规则判断

补充说明：
- 盖章状态只能说明“这份文件走完了盖章申请支线”
- 不能直接推出“这份文件已经可以作为正式业务依据”

---

## 9. 历史留痕规则
当前冻结口径为：

1. 盖章申请需要保留历史记录
2. 同一文件允许多次申请盖章
3. 每次申请都应保留各自历史
4. 当前阶段不展开版本管理细节，只冻结“可重复申请、需保留历史”的原则
