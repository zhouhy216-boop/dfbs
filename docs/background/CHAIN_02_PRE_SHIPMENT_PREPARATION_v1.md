# Chain Spec

## Meta
- chain_id: CHAIN_02
- chain_name: 发货前准备 / 装机信息回填链
- version: v1
- status: frozen-draft
- purpose: 中间标准稿，用于后续汇总成项目背景文件
- scope: 仅描述合同评审通过后、发货前必须完成的装机信息回填与平台/SIM资源开通协作
- not_for: 代码现状、实现方案、页面设计、任务单
- note: 真正的生产准备不在系统内；系统内只承接发货前必须补齐的信息

- 发货前准备
  - aliases: 发货前准备阶段
  - note: 合同评审通过后的下游阶段

- 装机信息回填链
  - aliases: 机器号回填链, 平台/SIM回填链
  - note: 本链的标准名称

- 机器号
  - aliases: 无
  - note: 本链主轴；后续平台、SIM、发货、收费、售后尽量围绕机器号串联

  - aliases: 平台SN, 平台设备身份号

  - aliases: 卡号, ICCID
  - note: 用我司SIM时与机器号建立1对1关系

- 临时绑定
  - aliases: 发货前绑定, 未生效绑定

## Main Object
- object_name: 发货前准备信息承接记录
- source: 合同评审通过结果
- result_on_completed: 发货前所需机器号、平台、SIM相关关系补齐，可进入发货链
- result_on_signed: 绑定关系正式生效，并作为后续收费与售后定位依据
- note:
  - 本链强调的是发货前必须承接的信息结果
  - 当前不在项目背景中冻结其是否必须独立成表

## Trigger
- 合同评审通过后触发
- 仅在发货前准备阶段使用
- 若合同存在平台/SIM相关需求，则进入相应分支

## Roles
- 营企
  - actions: 协调, 确认平台/SIM需求, 复核信息
  - note: 本链主责协调人，不直接分配卡池资源

- 生产企画
  - actions: 查看评审结果, 回填机器号, 回填机器号与SIM卡对应关系
  - note: 真正的生产准备不在系统里；其在系统内的核心职责是结果回填

- 平台管理员
  - actions: 预分配SIM资源, 开通平台, 开通SIM, 录入机构代码/简称, 必要时录入平台侧SN号, 维护发货后变更
  - note: 卡池和平台侧资源责任人

- 物流管理员
  - actions: 发货前异常兜底处理
  - note:
    - 不是默认流程责任人，只能在发货前异常已影响发货准备时兜底处理
    - 兜底不改变原申请方 / 原维护方的主责任归属

## Main Flow
1. 合同评审通过
2. 营企确认平台/SIM需求
3. 平台管理员按需求预分配平台/SIM资源
4. 生产企画在发货前回填机器号
5. 若使用我司SIM，生产企画继续回填 `机器号 ↔ SIM卡ICCID`
6. 若使用我司平台，平台管理员完成平台侧处理并形成机构代码/简称
7. 若个别平台要求平台侧SN号，则平台主管按平台规则处理该SN号
8. 若使用我司SIM，平台管理员完成SIM卡开通
9. 满足发货条件后进入发货链
10. 客户签收后，绑定关系正式生效

## Exception Fallback Rule
- 仅在发货前发现机器号、平台、SIM相关关系缺失、不完整或明显异常，且已实际影响发货准备时，才触发异常兜底
- 物流管理员可在异常场景下进行临时兜底处理，以避免发货准备中断
- 物流管理员的兜底处理不改变原申请方 / 原维护方的主责任归属
- 异常兜底必须留痕
- 当前留痕至少包括：
  - 兜底原因
  - 兜底人
  - 兜底时间
  - 处理结果
- 异常兜底完成后，后续仍应由原责任方补确认
- 当前版本只冻结“允许异常兜底并要求留痕”的业务原则，不展开更细的流程与实现方式

## Branch Matrix
- 用我司平台 + 用我司SIM
  - 平台管理员预分卡
  - 生产企画回填机器号与 `机器号 ↔ ICCID`
  - 平台管理员开平台、开卡、形成机构代码/简称
  - 如平台要求，再处理平台侧SN号
  - 发货
  - 签收后正式生效

- 用我司平台 + 不用我司SIM
  - 生产企画回填机器号
  - 平台管理员开平台、形成机构代码/简称
  - 如平台要求，再处理平台侧SN号
  - 发货
  - 签收后正式生效

- 不用我司平台 + 用我司SIM
  - 平台管理员预分卡
  - 生产企画回填机器号与 `机器号 ↔ ICCID`
  - 平台管理员开卡
  - 发货
  - 签收后正式生效

- 不用我司平台 + 不用我司SIM
  - 生产企画回填机器号
  - 发货

## Permission Rule
- 营企负责协调与复核，不直接承担卡池分配责任
- 平台管理员负责平台/SIM资源准备、开通与发货后变更维护
- 生产企画必须完成机器号回填；使用我司SIM时还必须完成 `机器号 ↔ ICCID` 回填
- 物流管理员只能异常兜底找机器号，不能成为默认流程

## Hard Gate Rule
- 没有机器号，不允许发货
- 使用我司SIM但没有 `机器号 ↔ ICCID`，不允许发货
- 发货前关系仅为临时绑定
- 签收后关系才正式生效
- 平台费限免期与SIM费限免期的起算节点，均与合同约定的保修起算节点相同
- 保修起算节点当前可为：发货 / 发船 / 签收 / 调试完成
- 平台费限免期与SIM费限免期在合同中允许原样填写，但系统统一折算为月

## Relationship Rule
- 使用我司平台时，机器号必须与机构代码/简称建立关系
- 个别平台还可能要求记录平台侧SN号；该SN号不等于机器号
- 不使用我司平台时，机器号不参与平台关系，但仍用于售后
- 使用我司SIM时，`SIM卡 ↔ 机器号` 为1对1
- 后续收费尽量通过机器号把平台费和SIM费串联
- 由于存在坏卡、换卡、退卡、停用、启用等情况，系统不能只保留当前关系，必须区分临时与正式，并保留变更历史

## Completion Rule
- 平台管理员完成平台/SIM侧处理，且发货前必需回填信息补齐后，本链可结束并进入发货链
- 客户签收后，相关绑定关系正式生效

## Boundary Note
- 本链只承接合同评审通过后、发货前必须补齐的机器号、平台、SIM相关准备信息
- 发货执行、出库、运输、签收与回执归属发货链处理
- 机器、SIM及后续正式主数据承接口径以主数据链为准
- 本链不展开页面入口、表结构、历史展示和具体实现方式

## Derived Rules
- 机器号是本链主轴
- 机器号必须在发货前形成
- 使用我司平台时，机器号与机构代码/简称建立关系
- 个别平台场景下，还需记录平台侧SN号
- 使用我司SIM时，机器号与SIM卡ICCID建立1对1关系
- 平台和SIM都使用我司服务时，后续收费通过机器号联动
- 签收前的绑定不作为正式生效关系
- 坏卡更换、客户取消订单退回卡池、现场换卡等情况不会改变“签收后才正式生效”的规则
- 发货前的初始申请与回填以生产企画为主
- 发货后的平台 / SIM / 关系变更由平台主管承接维护
- 平台费限免期与SIM费限免期沿用合同约定口径，不在本链另行定义新口径
- 本链只承接发货前准备阶段必须补齐的信息事实，不展开具体系统入口和实现方式

## Open Points
- 发货前异常兜底的更细责任留痕与追认流程，当前不展开
- 发货后的平台 / SIM / 关系变更具体操作入口，当前不在本链展开
- 发货前准备信息承接记录的对象实现方式，当前不在项目背景中冻结
