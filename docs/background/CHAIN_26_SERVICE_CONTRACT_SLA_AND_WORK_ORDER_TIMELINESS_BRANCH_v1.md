# CHAIN_26_SERVICE_CONTRACT_SLA_AND_WORK_ORDER_TIMELINESS_BRANCH_v1

## 1. 文档定位
本链用于说明服务合同 / 服务补充合同中的服务时效约定，如何被维修工单引用，并形成工单时效记录与统计结果。

本链不是工单执行流程本身，也不是服务补充合同 / 协议生命周期管理本身，而是一条位于两者之间的考核支链。其核心作用是：

- 从当前有效的服务约定中提取可用于时效考核的依据
- 将该依据应用到维修工单
- 形成响应、到场、完修三类时效结果
- 输出达标 / 超时的记录与统计结果

本链当前只解决“按什么规则考核工单时效”这一背景问题，不延伸到绩效制度和系统实现细节。

## 2. 核心业务目标
本链的核心目标，是建立服务时效约定与维修工单执行之间的统一连接口径，使系统能够明确：

- 哪些工单需要纳入 SLA 时效考核
- 工单时效考核优先引用哪类服务约定
- 在没有当前有效服务补充合同 / 协议时，按什么标准考核
- 响应、到场、完修三类时效分别如何定义
- 工单时效结果最终形成什么样的记录与统计结果

本链关注的是服务履约是否达标，而不是工单具体怎么流转。

## 3. 适用范围
当前阶段，本链只纳入维修工单进行 SLA 时效考核。

本链暂不纳入以下工单或服务场景：
- 培训类工单
- 调试类工单
- 巡检类工单
- 移机类工单
- 其他非维修类工单

因此，本链当前的适用范围明确收敛为：维修工单的服务时效履约管理。

## 4. 考核依据来源
维修工单的 SLA 时效考核，优先引用当前有效的服务补充合同 / 协议中的服务时效约定。

若当前有效服务补充合同 / 协议中存在响应、到场、完修等服务时效约定，则工单应优先按该约定进行考核。

若不存在当前有效服务补充合同 / 协议，或虽存在协议但未约定相关服务时效，则工单按公司默认标准进行考核。

因此，本链中的考核依据采用两层逻辑：

1. 优先使用当前有效服务补充合同 / 协议中的服务时效约定
2. 如无可引用的当前有效服务约定，则回落到公司默认标准

本链不负责判断哪份服务补充合同 / 协议当前有效，该问题由服务补充合同 / 协议生命周期管理链负责。

## 5. 当前纳入考核的时效项
当前阶段，本链只冻结三类时效项：

1. 响应时效
2. 到场时效
3. 完修时效

这三类时效项用于反映维修工单在不同关键阶段的履约表现，是当前版本中的核心 SLA 考核内容。

当前版本不继续扩展更多时效项，以避免过早复杂化。

## 6. 响应时效定义
响应时效定义为：报修到受理。

其中：
- 起点为客户报修时间
- 终点为客服正式受理工单时间

响应时效用于反映客户报修后，系统是否在约定时限内完成受理动作。

本链当前冻结到“报修到受理”这一层，不进一步展开更细的报修来源差异、受理例外和特殊渠道口径。

## 7. 到场时效定义
到场时效定义为：报修到到场签到。

其中：
- 起点为客户报修时间
- 终点为服务人员首次到场签到时间

到场时效用于反映客户报修后，现场服务是否在约定时限内到达客户现场。

本链当前冻结到“报修到到场签到”这一层，不进一步展开多次上门、补签到或特殊现场情况的细则。

## 8. 完修时效定义
完修时效定义为：报修到完修。

其中：
- 起点为客户报修时间
- 终点为工单完修时间

完修时效用于反映客户报修后，该维修工单是否在约定时限内完成维修闭环。

本链当前冻结到“报修到完修”这一层，不进一步展开返修、拆单、并单和复杂维修路径下的细则。

## 9. 达标与超时的判断原则
维修工单的响应、到场、完修三类时效，均应分别与其适用的 SLA 标准进行比对。

若某一时效项在约定时限内完成，则该时效项达标。
若某一时效项超过约定时限仍未完成，则该时效项超时。

因此，同一张维修工单可以同时形成三类时效判断结果：
- 响应是否达标
- 到场是否达标
- 完修是否达标

本链当前只定义达标 / 超时的业务判断原则，不展开更复杂的分级评分、扣分规则和责任追溯制度。

## 10. 暂停、豁免与不纳入边界
暂停、豁免与不纳入场景，会影响 SLA 计算结果，但当前版本不展开具体判定细则。

本链当前只冻结以下边界原则：

- 工单执行过程中可能存在影响 SLA 结果的暂停场景、豁免场景或不纳入场景
- 这些场景的业务事实来源于工单执行过程
- 当前版本仅保留该类边界，不展开具体判断规则和计算方式
- 后续如需细化，应在工单流程细化专题中补充

因此，本链当前不将 SLA 理解为一条简单的“从报修到结束的绝对直线时间”，而是承认真实业务中存在影响时效计算结果的例外场景，只是暂不细化。

## 11. 考核结果的输出方式
当前阶段，SLA 考核结果先用于记录和统计，不直接进入绩效扣罚制度。

本链当前要求形成两类结果输出：

1. 单工单结果
每张维修工单应可形成响应、到场、完修三类时效结果，并判断其是否达标或超时。

2. 统计结果
基于单工单结果，系统应能进一步形成面向服务经理、客服、主管和管理层的时效统计结果，用于观察和分析工单履约表现。

本链当前只冻结“记录 + 统计”这一层，不将其直接延伸为绩效扣罚制度或薪酬管理规则。

## 12. 与现有业务链的关系
### 12.1 与 CHAIN_25 的关系
CHAIN_25 负责判断当前有效的服务补充合同 / 协议是谁。  
本链负责按当前有效协议中的时效约定，对维修工单进行时效考核。

因此，CHAIN_25 是本链的上游依据链。

### 12.2 与 CHAIN_06 的关系
CHAIN_06 负责工单受理、派单、接单、上门、维修、完修、回访等执行过程。  
本链不重写工单执行流程，只引用工单过程中的关键时间点进行时效判断。

因此，本链是工单执行结果的考核支链，而不是工单执行主链。

### 12.3 与 CHAIN_23 的关系
本链形成的 SLA 结果，可进入报表输出层，作为专题统计与分析内容。  
但本链本身不展开报表菜单、报表样式、灵活查询和页面设计。

因此，本链提供的是考核结果基础，不是报表设计方案。

## 13. 当前边界与不展开项
本链当前只定义服务合同 / SLA 履约与维修工单时效考核的业务口径，暂不展开以下内容：

- 不展开工单主流程本身
- 不展开服务补充合同 / 协议生命周期管理
- 不展开绩效扣罚制度
- 不展开薪酬、奖金或 HR 考核制度
- 不展开暂停、豁免、不纳入的细则判断
- 不展开页面、字段、权限、提醒和预警实现
- 不展开报表菜单、报表样式和导出模板
- 不展开复杂评分模型、等级模型和责任归因模型

本链只解决“维修工单按什么服务时效口径进行履约考核”这一背景问题。

## 14. 小结
服务合同 / SLA 履约与工单时效考核，是连接“服务约定”和“工单执行结果”的独立业务支链。

当前阶段，本链已冻结以下核心原则：

- 当前只纳入维修工单进行 SLA 时效考核
- 工单考核优先引用当前有效服务补充合同 / 协议中的时效约定
- 若没有当前有效服务约定，则按公司默认标准考核
- 响应时效定义为报修到受理
- 到场时效定义为报修到到场签到
- 完修时效定义为报修到完修
- 考核结果当前只用于记录与统计，不直接进入绩效扣罚制度
- 暂停、豁免、不纳入等影响 SLA 结果的场景当前仅保留边界，不展开细则

因此，本链的核心价值在于：将服务时效约定真正转化为可被工单引用、可被业务记录、可被管理统计的履约判断基础。
